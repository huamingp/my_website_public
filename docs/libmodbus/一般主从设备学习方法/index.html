一般主从设备学习方法

一、通用主从通信框架（协议无关）​

在任何主从式通信系统中，一次典型的事务都遵循以下过程：

1. 总体角色与交互时序​

  • 主机（Master）：主动发起所有通信，发完一个请求后立即转为接收，等待应答。
  • 从机（Slave）：永远处于被动监听状态，收到完整请求后才执行动作并返回应答。
  • 一次事务 = 主机请求 → 从机应答（或超时/异常）。下一个事务必须等当前事务结束。

2. 主机行为框架​

  1. 明确通信意图：确定目标从机、操作类型（读/写）、操作对象（地址）、数据或长度。
  2. 协议编码：按照协议规则，将意图打包成字节流（含地址、命令、数据、校验）。
  3. 发送：满足总线空闲条件后，将帧完整发出，并控制字节间时序。
  4. 等待应答：启动超时定时器，等待从机返回数据。
  5. 接收与验证：收到应答帧后，校验完整性，解析数据，完成事务。超时或校验错则报错/重试。

3. 从机行为框架（被动接收处理管道）​

这是一个固定的处理流水线：

  1. 帧同步（界定起始）：从空闲中识别一个新帧的开始。
  2. 帧接收与定界：逐字节存入缓冲区，同时判别帧是否完整结束，过程中任何字节间超时都视为断帧丢弃。
  3. 完整性校验：对帧内校验域进行校验（如 CRC），错误则丢弃，不响应。
  4. 寻址过滤：检查帧内地址是否匹配本机地址（或广播），不匹配则丢弃。
  5. 命令解析与合法性检查：解析功能码、数据域长度，判断命令是否支持、参数是否合法。
  6. 执行操作：根据命令读写内部数据对象。
  7. 构造应答：
    • 正常：按协议格式打包返回的数据或回显确认。
    • 异常：返回包含错误码的异常应答。
  8. 发送应答：满足总线空闲条件后，将应答帧完整发出。

二、将 Modbus RTU 填入上述框架​

现在我们把这个框架具体化为 Modbus RTU 的行为，您会看到协议细节是如何落地的。

1. 主机行为：Modbus RTU 的实例化​

明确意图 → 编码 → 发送

以“读从机 1 的保持寄存器 0x0010，读 1 个”为例：

  • 意图：从机=1，功能码=读保持寄存器(0x03)，起始地址=0x0010，数量=1。
  • 编码规则（RTU 特色）：
    • 地址 1 字节，功能码 1 字节。
    • 起始地址和数量各 2 字节，大端序（高字节在前）。
    • CRC16 校验，计算范围（地址+功能码+数据），结果 低字节在前 追加。
  • 发送：
    • 必须等待总线空闲 ≥ T3.5（帧间隔），然后连续发出所有字节，字节间空闲 < T1.5。
  • 等待应答：
    • 启动应用层超时定时器（如 200ms），超时未收到数据则事务失败。

2. 从机处理管道：Modbus RTU 的实例化​

① 帧同步（识别帧头） RTU 报文没有帧头字符，完全靠时间：

  • 从机监测到总线空闲时间 ≥ T3.5 后，收到的第一个字节即为新帧的地址域。

② 帧接收与定界（T1.5 与 T3.5 协同）

  • 收到第一个字节后，启动 T1.5 字节间隔定时器。
  • 每收一个新字节，重装 T1.5 定时器。
  • 若 T1.5 超时（两个字节间隔过长）→ 帧不完整，丢弃。
  • 同时用 T3.5 帧间隔判断帧尾：收到一个字节后，若总线空闲时间 ≥ T3.5，则认为当前帧完全接收。

③ 完整性校验（CRC）

  • 对接收到的地址、功能码、数据域计算 CRC，与帧末两字节比对。
  • 不匹配 → 丢弃，不响应。

④ 寻址过滤

  • 检查地址是否等于本机地址或为 0（广播）。
    广播只执行写操作，不返回应答。不匹配则丢弃。

⑤ 命令解析与合法性检查

  • 功能码 0x03 合法，数据域长度应为 4 字节（地址2 + 数量2）。
  • 检查起始地址和数量是否在有效范围，否则转异常处理。

⑥ 执行操作

  • 从内部保持寄存器区读出 0x0010 的值，假设为 0x01F4。

⑦ 构造应答

  • 正常应答帧结构：
    • 地址、功能码、字节计数（寄存器数量×2）、寄存器值序列（每个值大端）。
    • 重新计算 CRC 追加。
  • 示例：01 03 02 01 F4 xx xx

⑧ 发送应答

  • 等待总线空闲 ≥ T3.5 后，连续发送应答帧。

三、Modbus RTU 的特色细节补充​

以上框架展示了 Modbus RTU 如何运行，下面总结其独有的协议要素，也是实现时必须掌握的知识点。

1. 数据模型（四种对象）​

从机内部数据被抽象为四种类型，各有不同的访问属性：

  • 线圈（Coils）：可读写的位（0x01 读，0x05 写单，0x0F 写多）
  • 离散输入（Discrete Inputs）：只读位（0x02 读）
  • 保持寄存器（Holding Registers）：可读写的 16 位字（0x03 读，0x06 写单，0x10 写多）
  • 输入寄存器（Input Registers）：只读 16 位字（0x04 读）

2. 常用功能码​

功能码名称操作对象
0x01读线圈位
0x02读离散输入位
0x03读保持寄存器16位字
0x04读输入寄存器16位字
0x05写单线圈位
0x06写单寄存器16位字
0x0F写多线圈位
0x10写多寄存器16位字

3. RTU 帧格式与字节序​

通用帧：地址（1B）+ 功能码（1B）+ 数据（N 字节）+ CRC（2B）

  • 多字节整数字节序：一律大端（高字节在前）。寄存器地址、数量、数据值均如此。
  • CRC 的字节序：小端。16 位 CRC 先发低字节，再发高字节。
  • 数据域具体结构因功能码而异：
    • 请求：通常包含起始地址（2B）和数量（2B）或写入值。
    • 响应：读操作返回“字节计数（1B）+ 数据”，写操作回显请求。

4. 时间间隔机制（RTU 独有）​

  • T1.5：帧内字节间隔上限。超过此时间，从机认为帧残缺。
    计算：T1.5 = 1.5 × 11 / 波特率 （当波特率 ≤ 19200）；
    当波特率 > 19200 时，固定为 750 µs。
  • T3.5：帧间间隔下限。新帧前、帧结束后总线必须空闲至少 T3.5。
    计算：T3.5 = 3.5 × 11 / 波特率；高波特率时固定为 1750 µs。

5. 异常处理机制​

当从机无法正常执行请求时，返回异常帧，格式为：
地址 + 功能码（原功能码 | 0x80） + 异常码 + CRC

常见异常码：

  • 0x01：非法功能
  • 0x02：非法数据地址
  • 0x03：非法数据值
  • 0x04：从机设备故障

主机收到异常帧后，根据异常码做相应处理（重试、报警、忽略等）。
