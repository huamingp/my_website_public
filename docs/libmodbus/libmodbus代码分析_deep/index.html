libmodbus 代码框架深度分析

以 modbus_write_bits（写多个线圈）和 modbus_read_registers（读保持寄存器）为例，分析 libmodbus RTU 模式下的代码架构与完整通信流程。

一、架构概览​

1.1 分层设计​

libmodbus 采用**后端抽象层（backend abstraction）**设计，将协议无关逻辑与物理层（RTU/TCP）解耦：

┌─────────────────────────────────────────────────┐
│  公开 API 层 (modbus.h)                          │
│  modbus_read_registers / modbus_write_bits / ... │
├─────────────────────────────────────────────────┤
│  核心逻辑层 (modbus.c)                            │
│  send_msg / _modbus_receive_msg / check_confirmation │
│  read_io_status / read_registers                │
├─────────────────────────────────────────────────┤
│  后端抽象层 (modbus-private.h: modbus_backend_t)  │
│  build_request_basis / send_msg_pre / send /     │
│  recv / check_integrity / pre_check_confirmation │
├─────────────────────────────────────────────────┤
│  物理层实现 (modbus-rtu.c / modbus-tcp.c)         │
│  CRC16 / write() / read() / select()            │
└─────────────────────────────────────────────────┘

1.3 消息类型枚举——主/从机视角​

（modbus-private.h:54-59）

typedef enum {

    MSG_CONFIRMATION   // 主机端：收到从机的响应
} msg_type_t;

Modbus 协议本质上是请求-响应模型：

  ---------- Request     Indication ----------
  | Client | ---------->| Server |
  ---------- Confirmation  Response ----------

同样的消息在线路上只有一个字节流。libmodbus 通过 msg_type 参数让同一个 _modbus_receive_msg 函数在两种角色下表现出不同行为。本文分析的是主机端——所有调用都传 MSG_CONFIRMATION。

二、modbus_write_bits—写多个线圈完整流程​

2.1 函数签名与入口校验​

modbus.h:227 声明，modbus.c:1423 实现：

int modbus_write_bits(modbus_t *ctx, int addr, int nb, const uint8_t *src);

参数：ctx（Modbus 上下文）、addr（起始线圈地址）、nb（线圈数量）、src（线圈值数组，每个元素为 TRUE/FALSE）。

modbus.c:1433-1447：

if (ctx == NULL) {
    errno = EINVAL;
    return -1;
}

if (nb > MODBUS_MAX_WRITE_BITS) {   // MODBUS_MAX_WRITE_BITS = 1968
    if (ctx->debug) {
        fprintf(stderr, "ERROR Writing too many bits (%d > %d)\n",
                nb, MODBUS_MAX_WRITE_BITS);
    }
    errno = EMBMDATA;
    return -1;
}

EMBMDATA（modbus.h:150）：

#define EMBMDATA (EMBXGTAR + 2)  // = MODBUS_ENOBASE + 12

2.2 第一步：构建请求帧​

写多个线圈使用功能码 MODBUS_FC_WRITE_MULTIPLE_COILS = 0x0F。请求 PDU 由两部分组成：固定头部 + 可变位打包数据。

2.2.1 build_request_basis—构建 PDU 头部​

modbus.c:1449-1450：

req_length = ctx->backend->build_request_basis(
    ctx, MODBUS_FC_WRITE_MULTIPLE_COILS, addr, nb, req);

modbus_t 中的 backend 字段指向编译期绑定的后端实例。modbus_new_rtu() 创建时绑定 _modbus_rtu_backend，因此此处实际调用 _modbus_rtu_build_request_basis（modbus-rtu.c:91-103）：

static int _modbus_rtu_build_request_basis(
    modbus_t *ctx, int function, int addr, int nb, uint8_t *req)
{
    assert(ctx->slave != -1);        // 从站地址必须在 modbus_set_slave() 中设置
    req[0] = ctx->slave;             // 字节0: 从站地址
    req[1] = function;               // 字节1: 功能码 0x0F
    req[2] = addr >> 8;             // 字节2: 起始地址高字节
    req[3] = addr & 0x00ff;         // 字节3: 起始地址低字节
    req[4] = nb >> 8;               // 字节4: 线圈数量高字节
    req[5] = nb & 0x00ff;           // 字节5: 线圈数量低字节

    return _MODBUS_RTU_PRESET_REQ_LENGTH;  // 返回 6
}

返回的 _MODBUS_RTU_PRESET_REQ_LENGTH = 6（modbus-rtu-private.h:23），即 PDU 头部固定为 6 字节：从站地址(1B) + 功能码(1B) + 起始地址(2B) + 数量(2B)。

2.2.2 位打包——线圈数据的压缩​

写线圈与写寄存器在 payload 构建上有本质区别。Modbus 协议中线圈是布尔量（0 或 1），每个线圈仅占 1 个比特，8 个线圈压缩为 1 字节。这是协议的基础要求，不是 libmodbus 的设计选择。

modbus.c:1451-1469：

byte_count = (nb / 8) + ((nb % 8) ? 1 : 0);   // 向上取整：计算需要多少字节
req[req_length++] = byte_count;                 // 字节计数字段紧跟 PDU 头

例如：nb = 3 → byte_count = (3 / 8) + 1 = 0 + 1 = 1 字节。nb = 10 → byte_count = 1 + 1 = 2 字节。

for (i = 0; i < byte_count; i++) {
    int bit;
    bit = 0x01;                        // 从最低位开始
    req[req_length] = 0;               // 当前字节初始化为 0

    while ((bit & 0xFF) && (bit_check++ < nb)) {
        if (src[pos++])                // TRUE → 该位写 1
            req[req_length] |= bit;
        else                           // FALSE → 该位写 0
            req[req_length] &= ~bit;

        bit = bit << 1;                // 左移到下一个比特位
    }
    req_length++;
}

while 循环的双重退出条件：

  • (bit & 0xFF)：限制一个字节内最多处理 8 个 bit。当 bit 左移到 0x100（第 9 位）时，bit & 0xFF 结果为 0，循环退出。
  • bit_check++ < nb：确保不会处理超出用户指定的线圈数量。

两个计数器各司其职：pos 追踪 src 数组的消费位置（无论跨几个字节都连续递增），bit_check 追踪已处理的线圈总数。

以 src = {TRUE, FALSE, TRUE}（3 个线圈）为例：

i=0, 初始 req[7] = 0x00
  bit_check=0, bit=0x01: src[0]=TRUE  → req[7] |= 0x01 → 0b00000001, bit=0x02, pos=1
  bit_check=1, bit=0x02: src[1]=FALSE → req[7]保持不变,        bit=0x04, pos=2
  bit_check=2, bit=0x04: src[2]=TRUE  → req[7] |= 0x04 → 0b00000101, bit=0x08, pos=3
  bit_check=3 >= nb=3 → 退出 while
req_length = 8 (从 7 递增到 8)

得到的请求帧（尚未添加 CRC）：

[01] [0F] [00] [10] [00] [03] [01] [05]
 从站  0x0F  起始地址=0x0010  数量=3  1字节  数据=0x05
                                      字节数   (bit0=1,bit1=0,bit2=1)

2.3 第二步：发送消息​

modbus.c:1471：

rc = send_msg(ctx, req, req_length);

2.3.1 CRC16 计算​

send_msg（modbus.c:165-221）第一步调用 ctx->backend->send_msg_pre。RTU 下为 _modbus_rtu_send_msg_pre（modbus-rtu.c:139-149）：

static int _modbus_rtu_send_msg_pre(uint8_t *req, int req_length)
{
    uint16_t crc = crc16(req, req_length);   // 对整个报文计算 CRC16

    // 低字节在前（小端序），符合 Modbus RTU 规范（p.14）
    req[req_length++] = crc & 0x00FF;
    req[req_length++] = crc >> 8;

    return req_length;  // 原长度 + 2
}

crc16 函数（modbus-rtu.c:116-130）：

static uint16_t crc16(uint8_t *buffer, uint16_t buffer_length)
{
    uint8_t crc_hi = 0xFF;  // CRC 寄存器初始化为 0xFFFF
    uint8_t crc_lo = 0xFF;
    unsigned int i;

    while (buffer_length--) {
        i = crc_lo ^ *buffer++;              // 当前字节 XOR CRC 低字节 → 查表索引
        crc_lo = crc_hi ^ table_crc_hi[i];  // 更新 CRC 低字节
        crc_hi = table_crc_lo[i];           // 更新 CRC 高字节
    }
    return (crc_hi << 8 | crc_lo);          // 合并为 16 位结果
}

查表法的两个预计算表 table_crc_hi[256]（modbus-rtu.c:30-49）和 table_crc_lo[256]（modbus-rtu.c:51-87），覆盖了所有可能的 8 位输入组合。多项式为 0x8005（CRC-16/MODBUS）。初始值 0xFFFF，不异或输出。

2.3.2 物理发送与错误恢复​

modbus.c:178-213，核心发送循环：

do {
    rc = ctx->backend->send(ctx, msg, msg_length);
    if (rc == -1) {
        _error_print(ctx, NULL);
        if (ctx->error_recovery & MODBUS_ERROR_RECOVERY_LINK) {
            if ((errno == EBADF || errno == ECONNRESET || errno == EPIPE)) {
                // 链路断开 → 关闭连接 → 等待 → 重连
                modbus_close(ctx);
                _sleep_response_timeout(ctx);
                modbus_connect(ctx);
            } else {
                // 其他错误 → 等待 → 清空缓冲区
                _sleep_response_timeout(ctx);
                modbus_flush(ctx);
            }
        }
    }
} while ((ctx->error_recovery & MODBUS_ERROR_RECOVERY_LINK) && rc == -1);

error_recovery 标志（modbus.h:174-178）：

typedef enum {
    MODBUS_ERROR_RECOVERY_NONE     = 0,
    MODBUS_ERROR_RECOVERY_LINK     = (1 << 1),  // 链路层自动恢复
    MODBUS_ERROR_RECOVERY_PROTOCOL = (1 << 2)   // 协议层自动恢复
} modbus_error_recovery_mode;

默认值为 MODBUS_ERROR_RECOVERY_NONE，do...while 条件不满足，发送失败立即返回 -1。用户可通过 modbus_set_error_recovery() 启用自动恢复。

发送完成后长度校验（modbus.c:215-218）：

if (rc > 0 && rc != msg_length) {
    errno = EMBBADDATA;
    return -1;
}

2.3.3 完整发送帧​

经过 send_msg_pre 添加 CRC 后：

[01] [0F] [00] [10] [00] [03] [01] [05] [CRC_LO] [CRC_HI]
 从站  功能  起始地址=16    数量=3   字节数  数据    CRC-16/MODBUS
                                       =1    =0x05    (小端序)

2.4 第三步：接收响应​

modbus.c:1475：

rc = _modbus_receive_msg(ctx, rsp, MSG_CONFIRMATION);

_modbus_receive_msg（modbus.c:350-521）是 libmodbus 最核心的接收引擎，通过三步状态机逐步解析消息。代码注释明确标注 "3 steps are used to parse the query"（modbus.c:37-42）：

typedef enum {
    _STEP_FUNCTION,    // 第一步：读到功能码
    _STEP_META,        // 第二步：读元数据（地址、数量、字节计数等）
    _STEP_DATA         // 第三步：读数据载荷与校验
} _step_t;

2.4.1 超时配置​

modbus.c:388-405，_modbus_receive_msg 同时服务于主机和从机，通过 msg_type 选择超时策略：

if (msg_type == MSG_INDICATION) {
    // 从机模式：被动监听，不知道主机何时发请求
    if (ctx->indication_timeout.tv_sec == 0 && ctx->indication_timeout.tv_usec == 0)
        p_tv = NULL;  // NULL → select() 无限阻塞，直到有数据到达
    else
        p_tv = &indication_timeout;
} else {
    // 主机模式：已发送请求，必须在 response_timeout 内收到完整响应
    tv.tv_sec = ctx->response_timeout.tv_sec;    // 默认 0.5s
    tv.tv_usec = ctx->response_timeout.tv_usec;  // 即 _RESPONSE_TIMEOUT = 500000us
    p_tv = &tv;
}

本文分析的场景全部使用 MSG_CONFIRMATION（主机收响应），超时后 select() 返回 ETIMEDOUT，函数返回 -1。

2.4.2 _STEP_FUNCTION​

modbus.c:385-386：

step = _STEP_FUNCTION;
length_to_read = ctx->backend->header_length + 1;  // RTU: 1 + 1 = 2 字节

初始读取 header_length + 1 字节。RTU 下 header_length = 1，所以读 2 字节。这是能确定功能码所需的最小字节数：从站地址 + 功能码。

对于写线圈的正常响应，读到：

[01] [0F]    // 从站地址=1, 功能码=0x0F（非异常，异常时 bit7=1）

读满 2 字节后 length_to_read == 0，状态机进入转换逻辑（modbus.c:481-502）：

if (length_to_read == 0) {
    switch (step) {
    case _STEP_FUNCTION:
        // 根据刚收到的功能码，计算还需要读多少元数据
        length_to_read = compute_meta_length_after_function(
            msg[ctx->backend->header_length], msg_type);
        if (length_to_read != 0) {
            step = _STEP_META;
            break;                  // break 跳出 switch，继续 while(length_to_read != 0)
        }
        // 如果返回 0，则 fall-through 跳过 _STEP_META 直达 _STEP_DATA
    case _STEP_META:
        length_to_read = compute_data_length_after_meta(ctx, msg, msg_type);
        // 长度校验：防止读取超过最大 ADU
        if ((msg_length + length_to_read) > ctx->backend->max_adu_length) {
            errno = EMBBADDATA;
            return -1;
        }
        step = _STEP_DATA;
        break;
    default:
        break;
    }
}

关键设计：case _STEP_FUNCTION 的 break 只跳出 switch，不跳出 while。如果 compute_meta_length_after_function 返回 0（某些功能码没有元数据），则会 fall-through 到 _STEP_META 的逻辑。

2.4.3 compute_meta_length_after_function​

modbus.c:265-301，根据功能码和消息类型决定元数据长度：

static uint8_t compute_meta_length_after_function(int function, msg_type_t msg_type)
{
    int length;

    if (msg_type == MSG_INDICATION) {
        // 从机收到请求：按请求类型决定
        if (function <= MODBUS_FC_WRITE_SINGLE_REGISTER) {   // 0x01~0x06
            length = 4;   // 读请求/写单个：地址(2B) + 数量或值(2B)
        } else if (function == MODBUS_FC_WRITE_MULTIPLE_COILS ||
                   function == MODBUS_FC_WRITE_MULTIPLE_REGISTERS) {
            length = 5;   // 写多个：地址(2B) + 数量(2B) + 字节计数(1B)
        } else if (function == MODBUS_FC_MASK_WRITE_REGISTER) {
            length = 6;   // Mask Write: 地址(2B) + and_mask(2B) + or_mask(2B)
        } else if (function == MODBUS_FC_WRITE_AND_READ_REGISTERS) {
            length = 9;   // Write and Read: 读地址(2B)+读数量(2B)+写地址(2B)+写数量(2B)+字节数(1B)
        } else {
            length = 0;   // 0x07, 0x11: 没有额外元数据
        }
    } else {
        /* MSG_CONFIRMATION — 主机收到响应 */
        switch (function) {
        case MODBUS_FC_WRITE_SINGLE_COIL:           // 0x05
        case MODBUS_FC_WRITE_SINGLE_REGISTER:        // 0x06
        case MODBUS_FC_WRITE_MULTIPLE_COILS:          // 0x0F
        case MODBUS_FC_WRITE_MULTIPLE_REGISTERS:      // 0x10
            length = 4;   // 写操作响应：地址回显(2B) + 数量或值回显(2B)
            break;
        case MODBUS_FC_MASK_WRITE_REGISTER:          // 0x16
            length = 6;   // Mask Write 回显：地址(2B)+and_mask(2B)+or_mask(2B)
            break;
        default:
            length = 1;   // 读操作响应：字节计数(1B)
        }
    }
    return length;
}

对主机收到写线圈响应（MSG_CONFIRMATION, function=0x0F）：length = 4 字节（地址 2B + 数量 2B）。

为什么写和读的元数据长度不同？

  • 写操作的响应固定回显地址和数量，所以需要 4 字节元数据。
  • 读操作的响应格式为 [功能码][字节数][数据...]，元数据只有 1 字节（字节计数）。

2.4.4 _STEP_META​

读 4 字节：

[00] [10] [00] [03]
 起始地址=16    数量=3

读满 4 字节后 length_to_read == 0，fall-through 到 _STEP_META：

case _STEP_META:
    length_to_read = compute_data_length_after_meta(ctx, msg, msg_type);
    // 安全检查
    if ((msg_length + length_to_read) > ctx->backend->max_adu_length) {
        errno = EMBBADDATA;
        return -1;
    }
    step = _STEP_DATA;
    break;

2.4.5 compute_data_length_after_meta​

modbus.c:305-336：

static int compute_data_length_after_meta(modbus_t *ctx, uint8_t *msg, msg_type_t msg_type)
{
    int function = msg[ctx->backend->header_length];  // msg[1] = 功能码
    int length;

    if (msg_type == MSG_INDICATION) {
        // 从机收到请求：只有写请求才带数据载荷
        switch (function) {
        case MODBUS_FC_WRITE_MULTIPLE_COILS:           // 0x0F
        case MODBUS_FC_WRITE_MULTIPLE_REGISTERS:        // 0x10
            length = msg[ctx->backend->header_length + 5];  // 第 5 个字节是数据字节计数
            break;
        case MODBUS_FC_WRITE_AND_READ_REGISTERS:        // 0x17
            length = msg[ctx->backend->header_length + 9];  // 第 9 个字节是写数据字节计数
            break;
        default:
            length = 0;   // 读请求没有数据载荷
        }
    } else {
        /* MSG_CONFIRMATION — 主机收到响应 */
        if (function <= MODBUS_FC_READ_INPUT_REGISTERS ||  // 0x01~0x04（读操作）
            function == MODBUS_FC_REPORT_SLAVE_ID ||        // 0x11
            function == MODBUS_FC_WRITE_AND_READ_REGISTERS) // 0x17
        {
            length = msg[ctx->backend->header_length + 1];  // 从响应中获取字节计数
        } else {
            length = 0;   // 写操作响应：没有数据载荷，只有元数据回显
        }
    }

    length += ctx->backend->checksum_length;  // RTU: +2 (CRC)
    return length;
}

对写线圈响应（MSG_CONFIRMATION, function=0x0F）：0x0F 不在 0x01~0x04 范围内，走 else 分支，length = 0 + checksum_length = 2。只需再读 2 字节 CRC。

2.4.6 _STEP_DATA—完整性校验​

读 2 字节 CRC，接收完成。调用 ctx->backend->check_integrity 进行完整性校验（modbus.c:521）：

return ctx->backend->check_integrity(ctx, msg, msg_length);

RTU 下为 _modbus_rtu_check_integrity（modbus-rtu.c:349-384）：

static int _modbus_rtu_check_integrity(modbus_t *ctx, uint8_t *msg, const int msg_length)
{
    uint16_t crc_calculated;
    uint16_t crc_received;
    int slave = msg[0];

    // 第一步：从站地址过滤——不是发给我的消息直接忽略，不浪费算力计算 CRC
    if (slave != ctx->slave && slave != MODBUS_BROADCAST_ADDRESS) {
        if (ctx->debug) {
            printf("Request for slave %d ignored (not %d)\n", slave, ctx->slave);
        }
        return 0;  // 消息被过滤，后续调用者根据返回值 0 做特殊处理
    }

    // 第二步：重新计算 CRC16，与收到的 CRC 对比
    crc_calculated = crc16(msg, msg_length - 2);          // 对除 CRC 外的所有字节重新计算
    crc_received = (msg[msg_length - 1] << 8) | msg[msg_length - 2];  // 收到的 CRC（小端序）

    if (crc_calculated == crc_received) {
        return msg_length;   // CRC 正确，返回消息总长度
    } else {
        if (ctx->debug) {
            fprintf(stderr, "ERROR CRC received 0x%0X != CRC calculated 0x%0X\n",
                    crc_received, crc_calculated);
        }
        if (ctx->error_recovery & MODBUS_ERROR_RECOVERY_PROTOCOL) {
            _modbus_rtu_flush(ctx);  // 清空串口缓冲区
        }
        errno = EMBBADCRC;
        return -1;           // CRC 错误
    }
}

CRC 正确时返回消息总长度。完整响应帧：

[01] [0F] [00] [10] [00] [03] [CRC_LO] [CRC_HI]
 从站  功能  起始地址=16    数量=3    CRC-16/MODBUS

三步状态机对这帧的处理：

_STEP_FUNCTION: 读 [01][0F] (2B) → compute_meta=4B
_STEP_META:     读 [00][10][00][03] (4B) → compute_data=2B
_STEP_DATA:     读 [CRC_LO][CRC_HI] (2B) → check_integrity: CRC正确 → 返回 6

2.5 第四步：确认验证​

modbus.c:1479：

rc = check_confirmation(ctx, req, rsp, rc);

check_confirmation（modbus.c:553-709）接收三个参数：req（原始请求帧）、rsp（收到的响应帧）、rsp_length（_modbus_receive_msg 返回的消息长度）。核心逻辑分四个步骤。

2.5.1 pre_check_confirmation—验证从站身份​

modbus.c:560-569，由后端实现：

if (ctx->backend->pre_check_confirmation) {
    rc = ctx->backend->pre_check_confirmation(ctx, req, rsp, rsp_length);
    if (rc == -1) {
        if (ctx->error_recovery & MODBUS_ERROR_RECOVERY_PROTOCOL) {
            _sleep_response_timeout(ctx);
            modbus_flush(ctx);
        }
        return -1;
    }
}

RTU 下为 _modbus_rtu_pre_check_confirmation（modbus-rtu.c:325-343）：

static int _modbus_rtu_pre_check_confirmation(modbus_t *ctx,
                                              const uint8_t *req,
                                              const uint8_t *rsp,
                                              int rsp_length)
{
    // 响应中的从站地址必须等于请求中的从站地址
    // 例外：广播地址不期待响应，不校验
    if (req[0] != rsp[0] && req[0] != MODBUS_BROADCAST_ADDRESS) {
        if (ctx->debug) {
            fprintf(stderr, "The responding slave %d isn't the requested slave %d\n",
                    rsp[0], req[0]);
        }
        errno = EMBBADSLAVE;
        return -1;
    }
    return 0;
}

MODBUS_BROADCAST_ADDRESS = 0（modbus.h:76）。广播请求在 RTU 半双工链路上也可能收到回声帧，pre_check_confirmation 不阻止这种情况（因为 req[0] == 0 时条件不成立，直接返回 0），后续由 _modbus_rtu_receive 中的 confirmation_to_ignore 标志处理。

2.5.2 异常码检测​

modbus.c:574-592：

if (function >= 0x80) {
    if (rsp_length == (int)(offset + 2 + ctx->backend->checksum_length) &&
        req[offset] == (rsp[offset] - 0x80)) {
        // 确认异常码是针对本次请求的（功能码第7位置1 = 请求功能码 + 0x80）
        int exception_code = rsp[offset + 1];
        if (exception_code < MODBUS_EXCEPTION_MAX) {
            errno = MODBUS_ENOBASE + exception_code;
            // MODBUS_ENOBASE = 112345678，所以 errno 范围为 112345678~112345688
        } else {
            errno = EMBBADEXC;
        }
        _error_print(ctx, NULL);
    } else {
        errno = EMBBADEXC;
        _error_print(ctx, NULL);
    }
    return -1;
}

Modbus 异常响应格式固定为 5 字节 PDU：[异常功能码(1B)][异常码(1B)]。系统定义的异常码（modbus.h:119-132）：

MODBUS_EXCEPTION_ILLEGAL_FUNCTION      = 0x01  // 非法功能码
MODBUS_EXCEPTION_ILLEGAL_DATA_ADDRESS  = 0x02  // 非法数据地址
MODBUS_EXCEPTION_ILLEGAL_DATA_VALUE    = 0x03  // 非法数据值
MODBUS_EXCEPTION_SLAVE_OR_SERVER_FAILURE = 0x04
MODBUS_EXCEPTION_ACKNOWLEDGE           = 0x05
MODBUS_EXCEPTION_SLAVE_OR_SERVER_BUSY  = 0x06
MODBUS_EXCEPTION_NEGATIVE_ACKNOWLEDGE  = 0x07
MODBUS_EXCEPTION_MEMORY_PARITY         = 0x08
// ... 等

2.5.3 长度校验​

modbus.c:594-597：

rsp_length_computed = compute_response_length_from_request(ctx, req);

if ((rsp_length == rsp_length_computed ||
     rsp_length_computed == MSG_LENGTH_UNDEFINED) &&
    function < 0x80) { ... }

compute_response_length_from_request（modbus.c:129-162）根据原始请求帧推算理论响应长度。

对 MODBUS_FC_WRITE_MULTIPLE_COILS (0x0F)，落在 default 分支：

default:
    length = 5;  // PDU 部分：功能码(1) + 地址(2) + 数量(2) = 5
break;

return offset + length + ctx->backend->checksum_length;
// = 1 + 5 + 2 = 8 字节

若实际或理论长度不符则返回 EMBBADDATA。

2.5.4 逐字段比对​

modbus.c:603-672，对 MODBUS_FC_WRITE_MULTIPLE_COILS：

case MODBUS_FC_WRITE_MULTIPLE_COILS:
case MODBUS_FC_WRITE_MULTIPLE_REGISTERS:
    // 检验 1：响应地址必须回显请求地址
    if ((req[offset + 1] != rsp[offset + 1]) ||
        (req[offset + 2] != rsp[offset + 2])) {
        resp_addr_ok = FALSE;
    }
    // 检验 2：响应数量必须回显请求数量
    req_nb_value = (req[offset + 3] << 8) + req[offset + 4];
    rsp_nb_value = (rsp[offset + 3] << 8) | rsp[offset + 4];
    break;

注意这里的字节索引逻辑。RTU 下 offset = header_length = 1：

req: [slave(0)] [func(1)] [addr_H(2)] [addr_L(3)] [nb_H(4)] [nb_L(5)] ...
rsp: [slave(0)] [func(1)] [addr_H(2)] [addr_L(3)] [nb_H(4)] [nb_L(5)]

所以 req[offset+1] 即 req[2] = 地址高字节，rsp[offset+1] 即 rsp[2] = 响应地址高字节。

对写操作，resp_data_ok 变量保持初始值 TRUE，不参与实际校验——写响应不包含数据，所以只校验地址和数量的回显一致性。

2.5.5 返回结果​

modbus.c:674-676：

if ((req_nb_value == rsp_nb_value) && (resp_addr_ok == TRUE) && (resp_data_ok == TRUE)) {
    rc = rsp_nb_value;  // 一切正确，返回写入线圈数量
}

modbus_write_bits 返回此值给调用者。调用者获得写入数量（示例中为 3）。

三、modbus_read_registers—读保持寄存器完整流程​

3.1 调用链​

modbus.h:222 声明，modbus.c:1322 实现：

modbus_read_registers(ctx, addr, nb, dest)      modbus.c:1322（公开 API）
  └─ read_registers(ctx, 0x03, addr, nb, dest)  modbus.c:1276（static 核心函数）

modbus_read_registers 自身只做校验和委托，核心逻辑在 static 函数 read_registers 中。

3.2 入口校验​

modbus.c:1326-1343：

if (ctx == NULL) {
    errno = EINVAL;
    return -1;
}

if (nb > MODBUS_MAX_READ_REGISTERS) {  // = 125
    if (ctx->debug) {
        fprintf(stderr, "ERROR Too many registers requested (%d > %d)\n",
                nb, MODBUS_MAX_READ_REGISTERS);
    }
    errno = EMBMDATA;
    return -1;
}

status = read_registers(ctx, MODBUS_FC_READ_HOLDING_REGISTERS, addr, nb, dest);
return status;

与 modbus_write_bits 的校验模式一致。注意 read_registers 内部（modbus.c:1283-1292）会再次校验 nb——这是防御性编程的冗余校验，因为 read_registers 也被其他读寄存器 API（modbus_read_input_registers）直接调用。

3.3 第一步：构建请求帧​

modbus.c:1294 同样调用 _modbus_rtu_build_request_basis，参数不同：

req_length = ctx->backend->build_request_basis(
    ctx, MODBUS_FC_READ_HOLDING_REGISTERS, addr, nb, req);

功能码为 0x03，_modbus_rtu_build_request_basis 构建 6 字节 PDU 头：

[01] [03] [00] [10] [00] [03]
 从站  0x03  起始地址=0x10    数量=3

与写操作的本质区别：请求 PDU 到此为止。 读请求只需告知从机"从哪开始读、读多少"，不需要发送任何数据内容。因此没有字节计数、没有数据打包步骤。req_length 固定为 6 字节（加 CRC 后为 8 字节）。

3.4 第二步：发送​

与写操作完全相同：send_msg → _modbus_rtu_send_msg_pre（加 CRC） → _modbus_rtu_send（物理发送）。同一段代码，不做区分。

3.5 第三步：接收响应​

调用 _modbus_receive_msg(ctx, rsp, MSG_CONFIRMATION)（modbus.c:1301），同样是三步状态机，但每步计算的长度与写操作完全不同。

_STEP_FUNCTION​

读 2 字节：

[01] [03]    // 从站地址=1, 功能码=0x03

compute_meta_length_after_function(0x03, MSG_CONFIRMATION)​

0x03 不在写功能码（0x05, 0x06, 0x0F, 0x10, 0x16）之列，走 default 分支（modbus.c:295-296）：

default:
    length = 1;   // 读响应元数据只有 1 字节：字节计数

_STEP_META​

读 1 字节——字节计数字段。例如 3 个寄存器 × 2 字节 = 6 字节：

[06]    // 字节计数 = 6

compute_data_length_after_meta(0x03, MSG_CONFIRMATION)​

modbus.c:324-327：

if (function <= MODBUS_FC_READ_INPUT_REGISTERS ||  // 0x01~0x04 在此范围
    function == MODBUS_FC_REPORT_SLAVE_ID ||        // 0x11
    function == MODBUS_FC_WRITE_AND_READ_REGISTERS) // 0x17
{
    length = msg[ctx->backend->header_length + 1];  // 读刚收到的字节计数字段
}

msg[header_length + 1] 即 msg[1 + 1] = msg[2]，这就是 _STEP_META 读到的字节计数（6）。因此 length = 6 + checksum_length = 8。

_STEP_DATA​

读 8 字节：6 字节寄存器数据 + 2 字节 CRC：

[12] [34] [56] [78] [9A] [BC] [CRC_LO] [CRC_HI]
  |_reg0=0x1234_| |_reg1=0x5678_| |_reg2=0x9ABC_|  CRC-16/MODBUS

_modbus_rtu_check_integrity 重新计算 CRC16 并与收到的 CRC 对比，通过后返回消息总长度。

三步状态机读写对比：

阶段写线圈 (MSG_CONFIRMATION, 0x0F)读寄存器 (MSG_CONFIRMATION, 0x03)
_STEP_FUNCTION 读2 字节（从站 ID + 功能码）2 字节（相同）
compute_meta 返回4 字节1 字节
_STEP_META 读4 字节（地址回显 + 数量回显）1 字节（字节计数）
compute_data 返回0 + 2 = 2 字节字节计数 + 2 字节
_STEP_DATA 读2 字节（仅 CRC）N + 2 字节（N 字节数据 + CRC）

3.6 第四步：确认验证​

check_confirmation 对 0x03 功能码的比对逻辑：

compute_response_length_from_request（modbus.c:142-145）计算理论响应长度：

case MODBUS_FC_READ_HOLDING_REGISTERS:
case MODBUS_FC_READ_INPUT_REGISTERS:
    length = 2 + 2 * (req[offset + 3] << 8 | req[offset + 4]);
    //      ↑ ↑
    //      │ └── 每个寄存器 2 字节 × 请求数量
    //      └── 响应中"功能码(1B)+字节计数(1B)"
    break;

若 nb = 3：length = 2 + 2 * 3 = 8。完整帧含从站地址和 CRC：offset + length + checksum = 1 + 8 + 2 = 11。

逐字段比对（modbus.c:631-637）：

case MODBUS_FC_READ_HOLDING_REGISTERS:
case MODBUS_FC_READ_INPUT_REGISTERS:
    req_nb_value = (req[offset + 3] << 8) + req[offset + 4];  // 请求中的寄存器数量 = 3
    rsp_nb_value = (rsp[offset + 1] / 2);                      // 响应字节数 / 2 = 6/2 = 3
    break;

与写操作的差异：

  • resp_addr_ok 和 resp_data_ok 变量对读操作保持初始值 TRUE，不参与校验——因为读响应不包含地址字段（格式为 [功能码][字节数][数据...]），没有地址可校。
  • 校验点仅为：功能码一致 + 长度正确 + 请求寄存器数 == 响应字节数 / 2。

3.7 第五步：数据解包​

写操作没有此步骤——这是读操作独有的最终环节（modbus.c:1309-1314）：

offset = ctx->backend->header_length;   // RTU: offset = 1（跳过从站地址）

for (i = 0; i < rc; i++) {
    // 每个寄存器 = 2 字节大端序：高字节在前，低字节在后
    dest[i] = (rsp[offset + 2 + (i << 1)] << 8)
            |  rsp[offset + 3 + (i << 1)];
}

逐轮展开（offset = 1, rc = 3）：

i=0: (i<<1)=0
     dest[0] = (rsp[1+2+0] << 8) | rsp[1+3+0]
             = (rsp[3] << 8) | rsp[4]
             = (0x12 << 8) | 0x34 = 0x1234

i=1: (i<<1)=2
     dest[1] = (rsp[1+2+2] << 8) | rsp[1+3+2]
             = (rsp[5] << 8) | rsp[6]
             = (0x56 << 8) | 0x78 = 0x5678

i=2: (i<<1)=4
     dest[2] = (rsp[1+2+4] << 8) | rsp[1+3+4]
             = (rsp[7] << 8) | rsp[8]
             = (0x9A << 8) | 0xBC = 0x9ABC

offset + 2 的含义：offset = 1 跳过从站地址(1B)，+ 2 跳过功能码(1B)和字节计数字段(1B)，正好定位到数据区的起始。(i << 1) 即 i * 2，因为每个寄存器占 2 字节。

四、读写核心差异汇总​

4.1 请求帧构建​

modbus_write_bits (0x0F)modbus_read_registers (0x03)
PDU 头部6 字节6 字节（格式完全相同）
Payload有：字节计数(1B) + 位打包数据无：请求到头部即结束

4.2 发送​

完全相同。 send_msg → send_msg_pre（加 CRC16） → backend->send() → 物理写入。函数不区分读写请求。

4.3 接收（三步状态机）​

写线圈 (MSG_CONFIRMATION, 0x0F)读寄存器 (MSG_CONFIRMATION, 0x03)
_STEP_FUNCTION读 2B读 2B
compute_meta4（地址+数量回显）1（字节计数）
_STEP_META读 4B读 1B
compute_data0 + 2（仅 CRC）N + 2（N 字节数据 + CRC）
_STEP_DATA读 2B读 N+2 B

4.4 确认验证（check_confirmation）​

校验项写线圈 (0x0F)读寄存器 (0x03)
pre_check_confirmation✓（检验 rsp[0]==req[0]）✓（同上）
异常码（function >= 0x80）✓✓
长度校验✓（compute_response_length_from_request）✓（同上）
功能码一致✓✓
地址回显✓（req[2-3]==rsp[2-3]）不适用（响应不含地址）

4.5 收尾处理​

写线圈读寄存器
数据转换无需（响应仅回显确认）大端序解包：2 字节 → 1 个 uint16_t

五、架构设计要点​

  1. 后端抽象：所有物理层操作通过 modbus_backend_t 函数指针表间接调用。RTU/TCP 后端各自提供 const 实例（_modbus_rtu_backend / _modbus_tcp_backend），在 modbus_new_rtu() / modbus_new_tcp() 时绑定。

  2. 消息类型分发：_modbus_receive_msg 是唯一同时服务主/从的接收引擎。通过 msg_type（MSG_INDICATION / MSG_CONFIRMATION）影响：超时策略、元数据长度计算、数据载荷长度计算。三个可插拔的长度计算函数使得同一套接收逻辑适应所有 Modbus 功能码和角色。

  3. 三步状态机：_STEP_FUNCTION → _STEP_META → _STEP_DATA 逐层递进，无需一次分配最大缓冲区（260 字节）。compute_meta_length_after_function 和 compute_data_length_after_meta 是状态转换的决策中枢，两函数的 if (msg_type == MSG_INDICATION) / else 分支完整编码了主/从模式下的不同预期。

  4. 分层验证：校验分布在三个层次：

    • check_integrity（物理层）：CRC16 + 从站地址过滤（返回 0 表示消息被忽略）
    • pre_check_confirmation（链路层）：响应从站 ID 是否匹配请求从站 ID
    • check_confirmation（协议层）：功能码、异常码、帧长度、地址回显（写操作）、数量匹配（读操作）

  5. 读写非对称：写请求带数据、响应做回显；读请求不带数据、响应才是数据载体。这一根本差异决定了三步状态机在两种操作下计算出的每一步读取长度完全不同，也决定了 check_confirmation 的校验策略不同。
