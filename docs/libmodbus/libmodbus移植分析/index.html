libmodbus STM32+FreeRTOS 移植参考分析

基于 demo 目录下的 STM32H5 + FreeRTOS + libmodbus 工程，分析移植思路和具体实现方式。

一、总体移植思路​

1.1 核心策略：新增替代修改，双后端共存​

demo 的移植采用了最关键的策略——不修改原始的 modbus-rtu.c，而是新建一个 modbus-st-rtu.c 文件，在其中实现一套全新的、适用于 STM32 平台的 RTU 后端。

原始 libmodbus 的后端注册:
  modbus_new_rtu() → backend = &_modbus_rtu_backend    (POSIX实现)

demo 新增的后端注册:
  modbus_new_st_rtu() → backend = &_modbus_rtu_backend_uart  (STM32实现)

这样做的好处：

  • 原始代码完整保留，便于对比学习
  • 核心协议层 modbus.c 的修改量最小化，只替换与操作系统相关的调用
  • 两个后端可同时编译，分别用于不同的物理接口（UART2、UART4、USB-CDC）

1.2 三个改造层面的划分​

移植工作分为三个层面，每个层面修改范围不同：

层面文件改动程度说明
平台抽象层uart_device.h/c + 各 UART 实现全新创建统一 UART 操作接口，隔离 HAL 依赖
后端适配层modbus-st-rtu.c全新创建RTU 后端全部函数，使用平台抽象层实现
核心协议层modbus.c少量微调替换 POSIX 调用为 FreeRTOS 调用
头文件modbus-private.h, modbus-rtu-private.h, modbus-rtu.h局部修改数据结构适配，新增构造函数声明

二、详细实现分析​

2.1 UART 设备抽象层（最关键的设计模式）​

2.1.1 struct UART_Device 接口定义​

文件：uart_device.h

struct UART_Device {
    char *name;
    int (*Init)( struct UART_Device *pDev, int baud, char parity,
                 int data_bit, int stop_bit);
    int (*Send)( struct UART_Device *pDev, uint8_t *datas,
                 uint32_t len, int timeout);
    int (*RecvByte)( struct UART_Device *pDev, uint8_t *data, int timeout);
    int (*Flush)(struct UART_Device *pDev);
};

设计思想：用函数指针表实现面向对象的多态。每个物理 UART 提供自己的 Init、Send、RecvByte、Flush 实现，上层 libmodbus 通过同一个接口操作不同的物理通道。

四个方法各自的职责：

方法职责阻塞特性
Init配置波特率等参数，创建 RTOS 资源（队列、信号量），启动首次接收非阻塞
Send通过 DMA 发送数据，用信号量等待发送完成阻塞，带超时
RecvByte从 FreeRTOS 队列中取一个字节阻塞，带超时
Flush清空接收队列中所有未处理的字节非阻塞

2.1.2 设备注册与查找机制​

文件：uart_device.c

// 全局设备注册表
static struct UART_Device *g_uart_devices[] = {
    &g_uart2_dev,
    &g_uart4_dev,
    &g_usbserial_dev
};

// 按名称查找设备
struct UART_Device *GetUARTDevice(char *name) {
    for (i = 0; i < sizeof(g_uart_devices)/sizeof(g_uart_devices[0]); i++) {
        if (!strcmp(name, g_uart_devices[i]->name))
            return g_uart_devices[i];
    }
    return NULL;
}

modbus_new_st_rtu("uart2", ...) 通过传入字符串 "uart2" 找到对应的 g_uart2_dev，获得该设备的所有操作方法。这使得应用程序只需通过名称字符串即可选择不同的物理 UART。

2.1.3 具体 UART 实现：以 UART2 为例​

文件：usart.c（第 462-554 行）

每个 UART 设备需要定义自己的 UART_Device 结构体实例和相关函数。

（1）Init 函数 — 创建 RTOS 资源并启动接收：

int UART2_Rx_Start(struct UART_Device *pDev, int baud, char parity,
                    int data_bit, int stop_bit) {
    if (!g_xUART2_RX_Queue) {
        g_xUART2_RX_Queue = xQueueCreate(200, 1);       // 200 字节深度的队列
        g_UART2_TX_Semaphore = xSemaphoreCreateBinary(); // 发送完成信号量
        HAL_UARTEx_ReceiveToIdle_DMA(&huart2, g_uart2_rx_buf, 100); // 启动 DMA+IDLE 接收
    }
    return 0;
}

关键点：

  • 用 if (!g_xUART2_RX_Queue) 保证资源只创建一次，多个 modbus_new_st_rtu 指向同一物理 UART 时不会重复创建
  • xQueueCreate(200, 1) — 队列深度 200，每个元素 1 字节。这为接收帧提供了足够的缓冲
  • 接收 buff 大小为 100 字节，适合 Modbus RTU 的 256 字节 ADU 限制

（2）Send 函数 — DMA 发送 + 信号量同步：

int UART2_Send(struct UART_Device *pDev, uint8_t *datas,
                uint32_t len, int timeout) {
    HAL_UART_Transmit_DMA(&huart2, datas, len);
    if (pdTRUE == xSemaphoreTake(g_UART2_TX_Semaphore, timeout))
        return 0;
    else
        return -1;
}

流程：启动 DMA 发送 → 阻塞等待信号量（超时值由调用者传入） → HAL 发送完成中断里 xSemaphoreGiveFromISR 唤醒。

（3）RecvByte 函数 — 从队列逐字节取数据：

int UART2_GetData(struct UART_Device *pdev, uint8_t *pData, int timeout) {
    if (pdPASS == xQueueReceive(g_xUART2_RX_Queue, pData, timeout))
        return 0;
    else
        return -1;
}

每次调用只取一个字节。上层 _modbus_receive_msg() 在 while 循环中反复调用此函数来逐字节构建完整帧。

（4）Flush 函数 — 清空队列：

int UART2_Flush(struct UART_Device *pDev) {
    int cnt = 0;
    uint8_t data;
    while (1) {
        if (pdPASS != xQueueReceive(g_xUART2_RX_Queue, &data, 0))  // 超时 = 0，非阻塞
            break;
        cnt++;
    }
    return cnt;
}

（5）设备结构体注册：

struct UART_Device g_uart2_dev = {
    "uart2",
    UART2_Rx_Start,    // Init
    UART2_Send,        // Send
    UART2_GetData,     // RecvByte
    UART2_Flush        // Flush
};

2.1.4 USB CDC ACM 设备实现​

文件：ux_device_cdc_acm.c（第 386 行）

struct UART_Device g_usbserial_dev = {
    "usb",
    USBSerial_Init,
    USBSerial_Send,
    USBSerial_GetData,
    USBSerial_Flush

USB 虚拟串口使用与硬件 UART 完全相同的 UART_Device 接口，体现了抽象层的威力。应用程序不需要知道底层是硬件 UART 还是 USB CDC——modbus_new_st_rtu("usb", ...) 即可。

2.2 核心协议层 modbus.c 的改造​

文件：modbus.c

核心协议层的修改遵循 最小替代原则，只替换直接依赖操作系统的调用。

2.2.1 头文件替换​

// 移除:
// #include <errno.h>
// #include <unistd.h>

// 新增:
#include "FreeRTOS.h"
#include "task.h"
#include "errno.h"  // 使用移植后的 errno

2.2.2 sleep 函数替换​

原始代码用 nanosleep() 实现微秒级延时，demo 改为 FreeRTOS 的 vTaskDelay()：

static void _sleep_response_timeout(modbus_t *ctx) {
    // 原来:
    // struct timespec request, remaining;
    // while (nanosleep(&request, &remaining) == -1 && errno == EINTR) { ... }

    // 改为:
    vTaskDelay(ctx->response_timeout.tv_sec / 1000
             + ctx->response_timeout.tv_usec * 1000);
}

注意：vTaskDelay 的精度是 RTOS tick（通常 1ms），对于 error recovery 场景的延时已经足够。

2.2.3 _modbus_receive_msg() 的核心改造​

这是最重要的修改点。原始代码依赖 select() + read() 两步操作来接收数据：

原始流程：

while (length_to_read != 0) {
    select(s, &rset, tv);     // 等待数据就绪或超时
    read(s, msg, length);      // 读取数据
}

demo 改造后的流程：

while (length_to_read != 0) {
    rc = backend->recv(ctx, msg, length, timeout_ms);  // 合并select+read为一次调用
}

具体代码（modbus.c: 409）：

// 原来:
// rc = ctx-> backend-> select(ctx, &rset, p_tv, length_to_read);
// rc = ctx-> backend-> recv(ctx, msg + msg_length, length_to_read);

// 改为: 合并为一次调用，传入超时时间
rc = ctx->backend->recv(ctx, msg + msg_length, length_to_read,
                        p_tv->tv_sec * 1000 + p_tv->tv_usec / 1000);

这意味着 select 和 recv 被合并为一次操作。在 modbus-st-rtu.c 的 _modbus_rtu_recv() 中，xQueueReceive 本身就具备 "阻塞等待+获取数据" 的双重功能，完美替代了 select+read 的组合。

同时，_modbus_rtu_select() 被简化为空函数（直接返回 0），不再需要实现：

static int _modbus_rtu_select(modbus_t *ctx, fd_set *rset, struct timeval *tv,
                               int length_to_read) {
    return 0;  // 功能已被 recv 内部的队列超时替代
}

2.2.4 printf 替换​

所有 printf / fprintf 调用替换为 debug_printf / debug_fprintf（宏或函数），这在嵌入式环境下可以重定向到串口输出或完全禁用：

if (ctx->debug) {
    debug_printf("[%.2X]", msg[i]);        // 原为 printf
}

2.2.5 错误恢复的适当保留​

#ifdef _WIN32 分支保留不动，#else 分支（即 POSIX / 嵌入式路径）中的错误恢复逻辑被保留但简化：

#else
    int saved_errno = errno;
    if ((errno == EBADF || errno == ECONNRESET || errno == EPIPE)) {
        modbus_close(ctx);
        _sleep_response_timeout(ctx);    // 已改为 vTaskDelay
        modbus_connect(ctx);
    } else {
        _sleep_response_timeout(ctx);
        modbus_flush(ctx);               // 通过 UART_Device:: Flush() 实现
    }

2.4 modbus-st-rtu.c — STM32 RTU 后端完整实现​

文件：modbus-st-rtu.c

这是移植的核心文件，约 427 行，实现了所有 17 个后端函数指针。

2.4.1 后端函数表注册​

定义了一个新的全局后端表 _modbus_rtu_backend_uart（第 336-357 行），与原始 _modbus_rtu_backend 完全独立：

const modbus_backend_t _modbus_rtu_backend_uart = {
    _MODBUS_BACKEND_TYPE_RTU,
    _MODBUS_RTU_HEADER_LENGTH,       // 1
    _MODBUS_RTU_CHECKSUM_LENGTH,     // 2
    MODBUS_RTU_MAX_ADU_LENGTH,       // 256

    // 以下为 17 个函数指针:
    _modbus_set_slave,               // → 不变，纯逻辑
    _modbus_rtu_build_request_basis, // → 不变，纯协议
    _modbus_rtu_build_response_basis,// → 不变，纯协议
    _modbus_rtu_prepare_response_tid,// → 不变，纯协议
    _modbus_rtu_send_msg_pre,        // → 不变，纯 CRC 计算
    _modbus_rtu_send,                // ★ 改为通过 UART_Device 发送
    _modbus_rtu_receive,             // → 不变，调用 _modbus_receive_msg
    _modbus_rtu_recv,                // ★ 改为通过 UART_Device 接收
    _modbus_rtu_check_integrity,     // → 不变，CRC 查表算法
    _modbus_rtu_pre_check_confirmation, // → 不变，地址比较
    _modbus_rtu_connect,             // ★ 改为调用 UART_Device:: Init
    _modbus_rtu_is_connected,        // ★ 简化为始终返回 1
    _modbus_rtu_close,               // ★ 简化为空函数
    _modbus_rtu_flush,               // ★ 改为调用 UART_Device:: Flush

可以看到，17 个函数指针中只有 6 个需要平台适配（标 ★），其余 11 个是纯协议/算法函数。

2.4.2 构造函数 modbus_new_st_rtu()​

第 362-426 行：

modbus_t *modbus_new_st_rtu(const char *device, int baud, char parity,
                             int data_bit, int stop_bit) {
    // 1. 参数校验
    // 2. pvPortMalloc 分配 modbus_t 上下文 (替代 malloc)
    // 3. _modbus_init_common(ctx) 初始化通用字段
    // 4. ctx-> backend = &_modbus_rtu_backend_uart  绑定后端表
    // 5. GetUARTDevice(device) 按名称获取 UART 设备
    // 6. pvPortMalloc 分配 modbus_rtu_t 后端数据
    // 7. ctx_rtu-> dev = pdev  关联设备
    // 8. 保存 device/baud/parity/data_bit/stop_bit
    // 9. confirmation_to_ignore = FALSE
}

与原始 modbus_new_rtu 的核心区别：

  • 使用 pvPortMalloc 而非 malloc
  • 通过 GetUARTDevice(name) 获取设备，而非存储 "/dev/ttyS0" 路径等待后续 open()
  • 不配置 RTS 相关字段（RS-485 方向控制未实现）

2.4.3 六个平台适配函数的详细实现​

（1）_modbus_rtu_send（第 146-159 行）：

static ssize_t _modbus_rtu_send(modbus_t *ctx, const uint8_t *req, int req_length) {
    modbus_rtu_t *ctx_rtu = ctx->backend_data;
    struct UART_Device *pdev = ctx_rtu->dev;

    if (0 == pdev->Send(pdev, (uint8_t *)req, req_length, TIMEROUT_SEND_MSG))
        return req_length;    // 发送成功，返回发送字节数
    else {
        errno = EIO;
        return -1;
    }
}

TIMEROUT_SEND_MSG 定义为 1000（毫秒）。发送函数内部使用 DMA，阻塞等待信号量。

（2）_modbus_rtu_recv（第 184-197 行）：

static ssize_t _modbus_rtu_recv(modbus_t *ctx, uint8_t *rsp, int rsp_length,
                                 int timeout) {
    modbus_rtu_t *ctx_rtu = ctx->backend_data;
    struct UART_Device *pdev = ctx_rtu->dev;

    if (0 == pdev->RecvByte(pdev, rsp, timeout))
        return 1;    // 每次调用只收 1 字节
    else {
        errno = EIO;
        return -1;
    }
}

注意：每次调用只接收 1 个字节。参数 rsp_length 实际上没有被用来限制读取量，而是上层的 while 循环（在 _modbus_receive_msg() 中）通过 length_to_read 来控制需要读多少字节。

这种 "逐字节接收" 的方式是最简单的实现，也是与原始 POSIX read() 语义最接近的方式。代价是每个字节都要走一次函数调用链，但对于 Modbus RTU 的低数据量（最多 256 字节/帧）来说完全可以接受。

（3）_modbus_rtu_connect（第 266-276 行）：

static int _modbus_rtu_connect(modbus_t *ctx) {
    modbus_rtu_t *ctx_rtu = ctx->backend_data;
    struct UART_Device *pdev = ctx_rtu->dev;

    pdev->Init(pdev, ctx_rtu->baud, ctx_rtu->parity,
               ctx_rtu->data_bit, ctx_rtu->stop_bit);

    ctx->s = 1;  // 非零标记已连接（替代 original fd）
    return 0;
}

UART 初始化已经在 CubeMX 生成的 MX_USART2_UART_Init() 中完成（波特率、数据位、停止位、校验位）。但 UART_Device::Init() 又被调用一次来传入相同的参数。在 UART2_Rx_Start() 中可以看到实际只在首次调用时创建队列和启动 DMA 接收。

（4）_modbus_rtu_is_connected（第 279-282 行）：

static unsigned int _modbus_rtu_is_connected(modbus_t *ctx) {
    return 1;  // STM32 上无需动态连接检查
}

嵌入式场景下 UART 始终连接（硬件固定），不需要动态检测连接状态。

（5）_modbus_rtu_close（第 285-287 行）：

static void _modbus_rtu_close(modbus_t *ctx) {
    // 空函数，嵌入式场景无需像 POSIX 那样恢复 termios
}

（6）_modbus_rtu_free（第 324-332 行）：

static void _modbus_rtu_free(modbus_t *ctx) {
    if (ctx->backend_data) {
        vPortFree(((modbus_rtu_t *) ctx->backend_data)->device);
        vPortFree(ctx->backend_data);
    }
    vPortFree(ctx);
}

free() → vPortFree()，这是 FreeRTOS 的内存释放函数。

2.7 FreeRTOS 任务设计分析​

文件：app_freertos.c

2.7.1 任务架构​

demo 创建了 4 个任务（含默认任务和 3 个 Modbus 任务）：

void MX_FREERTOS_Init(void) {
    // 默认任务: StartDefaultTask
    osThreadNew(StartDefaultTask, NULL, &defaultTask_attributes);

    // UART2: 主机模式，读寄存器 + 写线圈
    xTaskCreate(CH1_UART2_ClientTask, "CH1_UART2_ClientTask",
                200, NULL, osPriorityNormal, NULL);

    // UART4: 从机模式，提供 10 个线圈+10 个寄存器
    xTaskCreate(CH2_UART4_ServerTask, "CH2_UART4_ServerTask",
                200, NULL, osPriorityNormal, NULL);
}

每个 Modbus 任务独立绑定一个物理 UART：

  • CH1_UART2_ClientTask → UART2（主机/client 模式，主动发起读写）
  • CH2_UART4_ServerTask → UART4（从机/server 模式，被动响应请求）

2.7.2 主机任务模式（Client Task）​

第 248-290 行：

static void CH1_UART2_ClientTask(void *pvParameters) {
    ctx = modbus_new_st_rtu("uart2", 115200, 'N', 8, 1);
    modbus_set_slave(ctx, 1);
    modbus_connect(ctx);

    for (;;) {
        rc = modbus_read_registers(ctx, 1, nb, &val);  // 读保持寄存器 1
        if (rc != nb) continue;                          // 失败则重试

        Draw_Number(0, 0, val, 0xff0000);               // 显示在 LCD 上

        vTaskDelay(2000);                                // 2 秒间隔
        modbus_write_bit(ctx, 0, level);                 // 写线圈 0
        level = !level;
    }
}

主机任务的典型模式：

  1. 创建上下文 → 设置从机地址 → 连接
  2. 循环中调用 libmodbus API（阻塞式，内部完成请求-应答全过程）
  3. 处理返回结果
  4. 延时后进入下一次轮询

2.7.3 从机任务模式（Server Task）​

第 179-245 行：

static void CH2_UART4_ServerTask(void *pvParameters) {
    ctx = modbus_new_st_rtu("uart4", 115200, 'N', 8, 1);
    modbus_set_slave(ctx, 1);
    mb_mapping = modbus_mapping_new_start_address(0,10, 0,10, 0,10, 0,10);
    modbus_connect(ctx);

    for (;;) {
        do {
            rc = modbus_receive(ctx, query);    // 等待主机请求
        } while (rc == 0);                       // 0 表示被过滤，继续等待

        if (rc == -1 && errno != EMBBADCRC) {
            continue;                            // 非 CRC 错误才跳过
        }

        modbus_reply(ctx, query, rc, mb_mapping); // 处理请求并应答

        // 修改映射数据，供下次读取
        vTaskDelay(1000);
        mb_mapping->tab_registers[1]++;
    }
}

从机任务的关键点：

  • modbus_receive(ctx, query) 是阻塞调用，等待主机请求
  • 返回 0 表示帧被过滤（地址不匹配或 CRC 错误导致静默丢弃），需要继续等待
  • EMBBADCRC 错误不需要断开连接，从机应继续监听
  • 可以在两次请求之间修改 mb_mapping 的数据，模拟传感器值变化

2.7.4 多设备通信的实现​

demo 没有展示 "一个主机轮询多个从机" 的场景，但根据 libmodbus API 可以实现：

// 主机轮询多个从机的伪代码:
modbus_set_slave(ctx, 1);
modbus_read_registers(ctx, addr, nb, buf1);  // 读从机 1

modbus_set_slave(ctx, 2);                     // 切换目标
modbus_read_registers(ctx, addr, nb, buf2);  // 读从机 2

modbus_set_slave(ctx, 3);
modbus_write_register(ctx, addr, val);        // 写从机 3

pre_check_confirmation 会自动校验返回帧的地址是否与 ctx->slave 一致。

2.8 DMA+IDLE 中断接收机制（RTU 帧定界的核心）​

这是 RTU 移植中最关键的硬件机制。文件：usart.c

2.8.1 启动 DMA+IDLE 接收​

// 在 UART2_Rx_Start() 中:
HAL_UARTEx_ReceiveToIdle_DMA(&huart2, g_uart2_rx_buf, 100);

HAL_UARTEx_ReceiveToIdle_DMA() 是 STM32 HAL 提供的扩展函数，它同时做两件事：

  1. DMA 传输：UART 每收到一个字节，DMA 自动将其存入 g_uart2_rx_buf
  2. IDLE 检测：当 UART 总线空闲（超过 1 帧字符时间无新数据），触发 IDLE 中断

2.8.2 IDLE 中断回调​

void HAL_UARTEx_RxEventCallback(UART_HandleTypeDef *huart, uint16_t Size) {
    if (huart == &huart2) {
        // Size = 本次 DMA 传输实际收到的字节数
        for (int i = 0; i < Size; i++) {
            xQueueSendFromISR(g_xUART2_RX_Queue,
                             (const void *)&g_uart2_rx_buf[i], NULL);
        }
        // 立即重新启动 DMA+IDLE 接收，准备收下一帧
        HAL_UARTEx_ReceiveToIdle_DMA(&huart2, g_uart2_rx_buf, 100);
    }
}

三种回调的分工：

回调触发条件用途
HAL_UARTEx_RxEventCallbackDMA 收满 100 字节 OR IDLE 检测主要接收路径，将数据推入队列
HAL_UART_RxCpltCallbackDMA 收满 100 字节（未启用 IDLE 的旧方式）备选路径
HAL_UART_ErrorCallback接收错误重新启动 DMA 接收，恢复监听

2.8.3 IDLE 检测与 T3.5 的关系​

对于 RTU 协议的 T3.5 帧间隔要求：

波特率T3.5 标准值1 字符时间IDLE 检测时间对比
1152001750 µs (固定值)~87 µs~87 µsIDLE < T3.5
192001750 µs (固定值)~520 µs~520 µsIDLE < T3.5
96003.65 ms~1.04 ms~1.04 msIDLE < T3.5

demo 的实现中 没有在 IDLE 中断后再加 T3.5 延时。对于主机模式（Client Task），帧接收是通过 xQueueReceive(..., timeout_ms) 逐字节取的，上层 _modbus_receive_msg() 中的 length_to_read 为零时就完成了接收，加上 CRC 校验，即使 IDLE 中断提前触发也不会造成数据错误——因为 CRC 校验会拒绝不完整的帧，而队列中剩余的字节会被 flush() 清除。

然而，对于从机模式（Server Task），这种简化可能在帧边界模糊的情况下（两个帧之间间隔太短）导致问题。在实际移植中应考虑在 IDLE 回调中加入一个软件定时器确认 T3.5 时间。

2.8.4 发送完成信号量​

void HAL_UART_TxCpltCallback(UART_HandleTypeDef *huart) {
    if (huart == &huart2) {
        xSemaphoreGiveFromISR(g_UART2_TX_Semaphore, NULL);
    }
    // 同理 huart4
}

DMA 发送完成后，在中断中释放信号量，唤醒阻塞在 xSemaphoreTake() 上的发送函数。

2.8.5 为什么选用信号量而非互斥量​

在注释中有一个重要说明（第 487 行）：

/* 等待一个信号量(为何不用 mutex? 因为在中断里 Give mutex 会出错) */

这是因为 FreeRTOS 的互斥量（mutex）不能在 ISR 中释放，而二进制信号量（binary semaphore）的 xSemaphoreGiveFromISR() 可以在中断中安全调用。这正是发送完成回调需要在 ISR 上下文中唤醒任务的需求决定的。

三、总结：demo 移植方案的优点与简化项​

3.1 设计优点​

  1. UART 设备抽象层设计精巧：函数指针表实现了接口与实现的分离，同一套 libmodbus 代码可以透明地操作 UART2、UART4、USB CDC 三种物理接口
  2. 核心协议层最小修改：modbus.c 中的 PDU 构建、三步状态机、应答校验等核心逻辑完全保留，只替换了 OS 依赖点
  3. CRC 查表算法完整复用：纯算法函数无需任何修改
  4. DMA+IDLE 中断的巧妙利用：利用 STM32 硬件特性实现了类 T3.5 的帧检测
  5. FreeRTOS 队列缓冲：解决了 DMA 接收与任务读取之间的异步数据传递

3.2 简化/未实现项​

简化项原 libmodbusdemo 做法影响

3.3 从 STM32H5 移植到 STM32F4 的注意事项​

demo 使用的是 STM32H5 系列，但移植方案对 STM32F4 同样适用：

  1. HAL 库兼容：HAL_UARTEx_ReceiveToIdle_DMA() 在 STM32F4 的 HAL 库中同样存在（需要较新版本的 HAL 库，STM32Cube_FW_F4 V1.26.0 及以上版本已包含）
  2. DMA 配置差异：F4 使用 DMA1/DMA2（Stream 和 Channel 概念），H5 使用 GPDMA，CubeMX 会自动生成正确的初始化代码，usart.c 中只需修改 CubeMX 生成的初始化部分
  3. 中断优先级：F4 的 NVIC 优先级位数只有 4 位（16 级优先级），注意设置合适的优先级（demo 中设为 5 对 F4 同样适用）
  4. uart_device 抽象层：无需修改，因为它只依赖 HAL 的通用 API 和 FreeRTOS 的队列/信号量
  5. modbus-st-rtu.c：完全无需修改，因为它只调用 UART_Device 接口
