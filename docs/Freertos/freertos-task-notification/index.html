任务通知

任务通知是 FreeRTOS 提供的“任务到任务（或 ISR 到任务）一对一信号通道”。

它比队列/信号量更轻量，因为通知信息直接放在目标任务的 TCB 内，不需要额外创建独立内核对象。

核心概念：通知值与通知状态​

任务通知最关键的两个成员在目标任务 TCB 中：

  • ulNotifiedValue[]：通知值（uint32_t）
  • ucNotifyState[]：通知状态（uint8_t）

说明：新版本支持通知数组（configTASK_NOTIFICATION_ARRAY_ENTRIES），默认至少为 1。多数工程只用索引 0。

(1) 通知值 ulNotifiedValue​

你可以把它理解为“随通知携带的 32-bit 数据槽”，常见用法有三类：

  1. 计数模式（类似轻量计数信号量）

    • 发送端每次 +1。
    • 接收端用 ulTaskNotifyTake() 取走计数。

  2. 事件位模式（类似轻量事件标志）

    • 发送端按位 OR：BIT0/BIT1/...。
    • 接收端按位判断并清位。

  3. 覆盖/写值模式（传 32-bit 数据）

    • 发送端直接写入一个值（可覆盖旧值）。
    • 接收端通过 xTaskNotifyWait() 读取。

(2) 通知状态 ucNotifyState​

通知状态用于表达“这条通知槽当前是否有待处理通知，以及任务是否在等待通知”。常见状态值（内核宏）如下：

  • taskNOT_WAITING_NOTIFICATION
    • 当前任务未在该通知槽上阻塞等待。
  • taskWAITING_NOTIFICATION
    • 当前任务已调用等待 API 并阻塞，等待通知到来。
  • taskNOTIFICATION_RECEIVED
    • 已存在 pending（待处理）通知，等待任务通过接收 API 消费。

可以把它当作“通知是否已到达、是否已被任务消费”的状态机标记。

(3) 什么是 pending（待处理）通知​

在本笔记里，pending（待处理）通知指的是：

  • 该通知槽的状态 ucNotifyState 已经是 taskNOTIFICATION_RECEIVED
  • 但接收任务还没有通过接收 API 完成消费（如 ulTaskNotifyTake / xTaskNotifyWait）

也就是说：通知已经“到达并挂起”，但还没被任务“取走处理”。

如何判断是否为 pending：

  • taskNOTIFICATION_RECEIVED：是 pending
  • taskWAITING_NOTIFICATION：不是 pending（这是“正在等通知”）
  • taskNOT_WAITING_NOTIFICATION：不是 pending（当前无待处理通知）

补充边界场景（容易混淆）：

  1. 当发送动作是 eSetValueWithoutOverwrite 时：

    • 若目标当前没有 pending，则写入成功并变为 pending。
    • 若目标当前已经 pending，本次发送返回 pdFAIL，不会产生新的 pending 条目。

  2. pending 不是“队列里有多条消息”：

    • 每个通知槽只有一个通知值和一个通知状态。
    • 新通知可能累积到同一个值（如 eIncrement/eSetBits），也可能覆盖旧值（eSetValueWithOverwrite）。

  3. pending 何时结束：

    • 接收任务成功调用接收 API 并完成消费后，状态回到非 pending（通常为 taskNOT_WAITING_NOTIFICATION）。

与队列、信号量、事件组对比（详细）​

维度任务通知队列信号量（Binary/Counting）事件组
通信关系一对一（固定目标任务）多对多多对多（同步语义）多任务等待同一位集合
是否要单独创建对象不需要（在任务 TCB 内）需要 xQueueCreate需要 xSemaphoreCreate...需要 xEventGroupCreate

ISR 支持完整支持 FromISR完整支持 FromISR完整支持 FromISR（互斥量除外）支持 SetBitsFromISR（常为延后处理）
内存开销最小相对较高（与长度和项大小有关）中等中等
运行开销最低一档中等低到中等中等
是否适合广播不适合（天然一对一）通过多队列实现广播不适合广播适合“多任务等同一事件位”
是否适合传结构体不适合最适合不适合不适合
典型用途ISR 快速唤醒单任务、轻量事件通知、32-bit 快速传值生产者-消费者数据流资源计数/事件同步多条件联合等待（AND/OR）
主要限制仅面向指定任务；值宽固定 32 位拷贝开销与容量管理成本不传业务数据不传业务数据，且位数有限

选型建议：

  • 目标是“唤醒某一个任务，且只需 32-bit 信息”：优先任务通知。
  • 目标是“传结构化数据”：优先队列。
  • 目标是“资源计数/令牌管理”：优先计数信号量。
  • 目标是“多个条件位组合等待”：优先事件组。

通知状态转换关系（结合你提的 3 种场景）​

场景 A：A 通知 B，但 B 此时不关心（未等待）​

  1. B 当前状态：taskNOT_WAITING_NOTIFICATION
  2. A 发送通知后：
    • B 的 ulNotifiedValue 按指定动作更新（加一/置位/覆盖）
    • B 的 ucNotifyState 置为 taskNOTIFICATION_RECEIVED
  3. B 不会立刻因“等待解除”而切换（因为它本来没阻塞在通知上）

场景 B：A 通知 B，B 正在等待​

  1. B 调用等待 API 后进入阻塞：taskWAITING_NOTIFICATION
  2. A 发送通知：
    • 更新 ulNotifiedValue
    • 状态置为 taskNOTIFICATION_RECEIVED
    • B 从 Blocked 转到 Ready
  3. 若 B 优先级更高，可能马上发生上下文切换

场景 C：A 先通知，B 之后才想“看之前有没有通知”​

  1. A 先发通知时，B 未等待，状态会变为 taskNOTIFICATION_RECEIVED
  2. B 后续可消费这条 pending 通知，但应按发送动作选择接收 API：
    • 计数语义（xTaskNotifyGive / eIncrement）优先用 ulTaskNotifyTake()
    • 位标志或写值语义（eSetBits / eSetValue...）优先用 xTaskNotifyWait()
  3. 消费后，状态通常回到 taskNOT_WAITING_NOTIFICATION

结论：任务通知在接收任务未阻塞时，通知状态仍可被置为已收到，不会“立刻失效”。 但任务通知不是队列：每个通知槽只有一个通知值与一个 pending 状态。 是否会覆盖、累积或发送失败，取决于发送动作（eAction）与接收侧清除策略。

内部机制（结合源码执行链路）​

以下用 FreeRTOS 内核常见实现路径说明（tasks.c，不同版本函数名细节可能有差异，但机制一致）。

(1) 发送主路径：xTaskGenericNotify() / xTaskGenericNotifyFromISR()​

发送通知时内核核心步骤：

  1. 定位目标任务 TCB 与通知索引（默认索引 0）。
  2. 先保存旧通知值（若调用方需要返回旧值）。
  3. 按 eAction 更新 ulNotifiedValue：
    • eNoAction：只改状态，不改值
    • eSetBits：value |= ulValue
    • eIncrement：value++
    • eSetValueWithOverwrite：value = ulValue（覆盖）
    • eSetValueWithoutOverwrite：若已有 pending 通知则失败，否则写入
  4. 将 ucNotifyState 置为 taskNOTIFICATION_RECEIVED。
  5. 若目标任务原本在该通知上阻塞（taskWAITING_NOTIFICATION），则将其从阻塞列表移到就绪列表。
  6. 在 ISR 版本中通过 pxHigherPriorityTaskWoken 告知是否需要在中断退出后触发调度。

这条链路解释了“通知值更新 + 状态更新 + 可能唤醒任务”是一次原子语义流程。

(2) 接收主路径一：ulTaskNotifyTake()（简化版）​

典型行为：

  1. 若通知值为 0 且允许等待：
    • 当前任务状态置 taskWAITING_NOTIFICATION
    • 挂入延时/等待链表并阻塞
  2. 被通知唤醒后读取当前 ulNotifiedValue
  3. 根据 xClearCountOnExit：
    • pdTRUE：退出时清零（常用于二值语义）
    • pdFALSE：退出时减一（常用于计数语义）
  4. ucNotifyState 回到 taskNOT_WAITING_NOTIFICATION

(3) 接收主路径二：xTaskNotifyWait()（专业版）​

典型行为：

  1. 进入等待前可先清位：ulBitsToClearOnEntry
  2. 若无 pending 通知且允许等待：置 taskWAITING_NOTIFICATION 并阻塞
  3. 收到通知后返回，输出当前通知值到 pulNotificationValue
  4. 退出前再按 ulBitsToClearOnExit 清位
  5. 最终将通知状态恢复为非等待态

它适合“位标志 + 可控清位”的高级场景。

(4) 为什么任务通知更轻量​

因为它不需要单独分配 Queue/Semaphore/EventGroup 控制块，直接使用目标任务 TCB 中已有字段：

  • 更少的 RAM 占用
  • 更短的内核路径
  • ISR 到任务唤醒延迟更低（常见场景下）

代价是：

  • 天然一对一，不适合多消费者广播
  • 数据宽度固定 32-bit

API 详解（参数类型、含义、返回值）​

发送通知：简化版​

  1. xTaskNotifyGive
BaseType_t xTaskNotifyGive( TaskHandle_t xTaskToNotify );
  • 参数
    • xTaskToNotify（TaskHandle_t）：目标任务句柄
  • 本质
    • 等价于 xTaskNotify(xTaskToNotify, 0, eIncrement) 的便捷封装
  • 返回值（BaseType_t）
    • 通常为 pdPASS
  • 语义
    • 目标通知值加一，常与 ulTaskNotifyTake() 配对
  1. vTaskNotifyGiveFromISR
void vTaskNotifyGiveFromISR( TaskHandle_t xTaskToNotify,
                                           BaseType_t *pxHigherPriorityTaskWoken );
  • 参数
    • xTaskToNotify（TaskHandle_t）：目标任务句柄
    • pxHigherPriorityTaskWoken（BaseType_t *）：若唤醒到更高优先级任务，内核会写 pdTRUE
  • 返回值
    • void
  • 使用要点
    • ISR 退出前调用 portYIELD_FROM_ISR(*pxHigherPriorityTaskWoken)

发送通知：专业版​

  1. xTaskNotify
BaseType_t xTaskNotify( TaskHandle_t xTaskToNotify,
                                    uint32_t ulValue,
                                    eNotifyAction eAction );

  • 参数

    • xTaskToNotify（TaskHandle_t）：目标任务
    • ulValue（uint32_t）：参与更新的值
    • eAction（eNotifyAction）：更新动作
      • eNoAction
      • eSetBits
      • eIncrement
      • eSetValueWithOverwrite
      • eSetValueWithoutOverwrite

  • 返回值（BaseType_t）

    • pdPASS：成功
    • pdFAIL：常见于 eSetValueWithoutOverwrite 且目标已有 pending 通知

  • eNotifyAction 各可选值的操作意义（重点）

eAction对 ulNotifiedValue 的操作典型语义什么时候用
eNoAction不改通知值，只把状态置为已收到纯“敲门通知”只关心“有事件来了”，不需要携带数值
eSetBits`ulNotifiedValue= ulValue`事件位累积（位标志）
eIncrementulNotifiedValue++计数累计ISR/任务多次触发，接收方按次数消费
eSetValueWithOverwriteulNotifiedValue = ulValue（无条件覆盖）最新值覆盖只关心最新状态/最新采样值
eSetValueWithoutOverwrite若当前无 pending 通知才写入；否则失败防覆盖写入不能接受新值覆盖旧值，要求发送端处理失败分支

为什么 eSetBits 是按位或（OR）​

eSetBits 的目标不是“替换旧值”，而是“把新事件位并入已有事件集合”。

  • 位或规则：某一位只要有一次被置 1，就保持为 1，直到接收方显式清位。
  • 这样可以实现“多事件累积不互相覆盖”。

示例：

// 约定：BIT0=DMA完成，BIT1=UART完成
// 初始通知值为 0b0000

// 先来 DMA 完成事件
xTaskNotify(task, (1u << 0), eSetBits);   // 值变为 0b0001

// 再来 UART 完成事件
xTaskNotify(task, (1u << 1), eSetBits);   // 值变为 0b0011

如果这里不用 OR，而是直接赋值覆盖，那么后一个事件会把前一个事件冲掉，接收任务就无法知道“两个事件都发生过”。

工程意义：

  1. 把 32-bit 通知值当作“轻量事件组”使用。
  2. 多个 ISR/任务可向同一目标任务汇报不同事件位。也就是可以用“某一位”表示“某个事件已经发生”。
    • 例子：BIT0 表示 DMA 完成，BIT1 表示 UART 接收完成。
    • DMA 中断到来时：xTaskNotifyFromISR(xTask, (1u << 0), eSetBits, &xHigherPriorityTaskWoken);
    • UART 中断到来时：xTaskNotifyFromISR(xTask, (1u << 1), eSetBits, &xHigherPriorityTaskWoken);
    • 这样通知值里哪一位是 1，就表示对应事件“至少发生过一次且尚未被清位”。
  3. 接收任务可一次性读取并按位分发处理，然后按需清位。也就是读取通知值后，按位判断“哪个事件发生了”，执行对应处理。
    • 例子：
    uint32_t ulEvents = 0;
    if (xTaskNotifyWait(0, ((1u << 0) | (1u << 1)), &ulEvents, portMAX_DELAY) == pdTRUE)
    {
       if ((ulEvents & (1u << 0)) != 0u) { HandleDmaDone(); }
       if ((ulEvents & (1u << 1)) != 0u) { HandleUartRxDone(); }
    }

eSetBits 常见误区与清位建议​

  1. 误区：把 eSetBits 当“覆盖写”

    • 错误理解：新事件来了就替换旧值。
    • 正确行为：eSetBits 是位累积（OR），旧位不会被自动清除。

  2. 误区：只置位不清位

    • 结果：同一事件位会长期为 1，后续逻辑可能反复误判“新事件又来了”。
    • 建议：接收端在处理完成后按位清除（例如用 xTaskNotifyWait 的 ulBitsToClearOnExit）。

  3. 推荐模板（位事件模式）

// BIT0: DMA done, BIT1: UART done
uint32_t ulEvents = 0;

if (xTaskNotifyWait(0, ((1u << 0) | (1u << 1)), &ulEvents, portMAX_DELAY) == pdTRUE)
{
   if ((ulEvents & (1u << 0)) != 0u)
   {
      HandleDmaDone();
   }

   if ((ulEvents & (1u << 1)) != 0u)
   {
      HandleUartDone();
   }
}

实战建议：事件位较多时，统一定义事件位宏与清位策略（入口清位/出口清位），避免位状态管理混乱。

  1. xTaskNotifyFromISR
BaseType_t xTaskNotifyFromISR( TaskHandle_t xTaskToNotify,
                                              uint32_t ulValue,
                                              eNotifyAction eAction,
                                              BaseType_t *pxHigherPriorityTaskWoken );
  • 参数
    • 前 3 个与任务态一致
    • pxHigherPriorityTaskWoken（BaseType_t *）：用于 ISR 退出调度请求
  • 返回值（BaseType_t）
    • 同任务态：pdPASS / pdFAIL

接收通知：简化版​

  1. ulTaskNotifyTake
uint32_t ulTaskNotifyTake( BaseType_t xClearCountOnExit,
                                        TickType_t xTicksToWait );
  • 参数
    • xClearCountOnExit（BaseType_t）
      • pdTRUE：取到通知后把通知值清零
      • pdFALSE：取到通知后通知值减一
    • xTicksToWait（TickType_t）：等待超时
      • 0：不等待
      • portMAX_DELAY：可长期等待（受配置影响）
  • 返回值（uint32_t）
    • 返回取到前的通知计数值
    • 返回 0 通常表示超时未收到
  • 典型配对
    • 与 xTaskNotifyGive / vTaskNotifyGiveFromISR 搭配构成轻量“计数信号量”模型

接收通知：专业版​

  1. xTaskNotifyWait
BaseType_t xTaskNotifyWait( uint32_t ulBitsToClearOnEntry,
                                          uint32_t ulBitsToClearOnExit,
                                          uint32_t *pulNotificationValue,
                                          TickType_t xTicksToWait );
  • 参数
    • ulBitsToClearOnEntry（uint32_t）：进入等待前先清除的位
    • ulBitsToClearOnExit（uint32_t）：成功收到通知后退出前清除的位
    • pulNotificationValue（uint32_t *）：输出通知值快照（可为 NULL）
    • xTicksToWait（TickType_t）：等待时长
  • 返回值（BaseType_t）
    • pdTRUE：收到通知
    • pdFALSE：超时未收到
  • 典型用途
    • 多 bit 事件协同、可控清位策略、读取并解析 32-bit 载荷

工程实践建议​

  1. ISR 快速唤醒单任务，优先“任务通知 + FromISR API”。
  2. 需要保留历史多条数据时，不要用任务通知硬凑，直接用队列。
  3. 使用 eSetValueWithoutOverwrite 时，必须处理 pdFAIL 分支，避免静默丢通知。
  4. 若任务长期不消费通知，taskNOTIFICATION_RECEIVED 会持续存在，需明确清除策略。
  5. 对于事件位模式，统一定义 BIT 宏并规范清位时机，避免“位粘连”导致误判。
