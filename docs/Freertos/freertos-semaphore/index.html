信号量

信号量可以先当成一个“带门禁的计数器”：

  • give：把计数器+1，表示“资源多了一个”或“事件来了一次”。
  • take：把计数器-1，表示“我要消耗一个资源”或“我处理了一个事件”。

当计数器是 0 时，take 会失败。此时是否阻塞，取决于 xTicksToWait：

  • 0：不等，立刻返回失败。
  • portMAX_DELAY：一直等（前提是系统允许无限等待）。
  • 其他值：最多等待指定 Tick 数，常用 pdMS_TO_TICKS(ms) 转毫秒。

一句话记忆：队列用于“传数据”，信号量用于“传时机”。

队列和信号量：异同对照表​

对比项队列（Queue）信号量（Semaphore）
核心目标传递数据 + 同步同步事件 / 管理资源数量
是否承载业务数据承载（拷贝数据项）不承载（只传“可否继续”）
常用场景生产者-消费者数据流ISR 通知任务、连接池/缓冲池配额
内存开销与队列长度和数据项大小相关通常更小
API 结果关注点收发是否成功 + 数据内容take/give 是否成功
是否能表达“有多少个资源”不是主要用途计数信号量天然支持

经验选型：

  • 你关心“把结构体/字节流传过去”：优先队列。
  • 你关心“通知一下、给个令牌”：优先信号量。

特性​

  • 不拷贝业务数据，只表达“现在能不能继续执行”，起到通知作用。
  • 支持阻塞等待，避免因为抢不到资源而空循环死等，空转抢 CPU。
  • 可用于“资源管理”（计数信号量）和“事件通知”（二值信号量）。
  • 任务态和 ISR 态都可用，但 ISR 必须使用 FromISR 版本 API。
  • 信号量 give 只负责“解除阻塞 + 状态迁移”：从该信号量等待列表中选出可唤醒任务（先看优先级，同优先级按等待顺序），把任务从阻塞态移入对应优先级就绪列表，并更新最高就绪优先级索引。
  • 在 ISR 中，若本次唤醒到更高优先级任务，会通过 pxHigherPriorityTaskWoken 配合 portYIELD_FROM_ISR() 请求一次 PendSV；若条件不满足则先不切换。
  • 真正的运行决策发生在调度阶段：调度器只在就绪集合中统一选择“最高优先级任务”运行；同优先级再按时间片与就绪链表规则轮转。

常见用途：

  • 二值信号量：DMA/ADC/UART 中断通知任务，（用来管理临界资源并不可靠）。
  • 计数信号量：表示 N 个同类资源可用（连接、缓冲块、设备通道）。

计数型信号量（Counting Semaphore）​

计数范围是 0 ~ uxMaxCount，可以理解为“资源余量表”。

创建​

动态创建：

SemaphoreHandle_t xSemaphoreCreateCounting( UBaseType_t uxMaxCount,
                                            UBaseType_t uxInitialCount );
  • uxMaxCount：计数上限，不能超过这个值。
  • uxInitialCount：初始可用数量。
  • 返回值：成功返回句柄；失败返回 NULL。

静态创建：

SemaphoreHandle_t xSemaphoreCreateCountingStatic( UBaseType_t uxMaxCount,
                                                  UBaseType_t uxInitialCount,
                                                  StaticSemaphore_t *pxSemaphoreBuffer );
  • pxSemaphoreBuffer：用户提供的控制块内存。
  • 适合禁用或减少运行期动态内存分配的工程。

删除​

void vSemaphoreDelete( SemaphoreHandle_t xSemaphore );
  • 动态创建对象：删除后释放其内核对象内存。
  • 静态创建对象：删除后对象失效，静态内存仍由用户管理。

Give / Take（任务态）​

BaseType_t xSemaphoreTake( SemaphoreHandle_t xSemaphore,
                           TickType_t xTicksToWait );

BaseType_t xSemaphoreGive( SemaphoreHandle_t xSemaphore );

参数解释：

  • xSemaphore：目标信号量句柄。
  • xTicksToWait：拿不到令牌时最多等待多久。=0时，不等待，直接返回，若=portMAX_DELAY，则阻塞等待直到返回成功。=其他值时，最多等待指定 Tick 数，常用 pdMS_TO_TICKS(ms) 转等待多少毫秒。

返回值解释：

  • xSemaphoreTake：成功 pdTRUE，超时/失败 pdFALSE。
  • xSemaphoreGive：成功 pdTRUE，若已达上限通常返回 pdFALSE。

行为理解：

  • 计数 > 0：take 立即成功，计数减一。
  • 计数 = 0 且允许等待：任务进入 Blocked，等待别人 give 或超时。

Give / Take（ISR 态）​

BaseType_t xSemaphoreTakeFromISR( SemaphoreHandle_t xSemaphore,
                                  BaseType_t *pxHigherPriorityTaskWoken );

BaseType_t xSemaphoreGiveFromISR( SemaphoreHandle_t xSemaphore,
                                  BaseType_t *pxHigherPriorityTaskWoken );

参数解释：

  • xSemaphore：目标信号量句柄。
  • pxHigherPriorityTaskWoken：输出参数。若本次操作让更高优先级任务从阻塞态变成就绪态，会被置为 pdTRUE。

常见写法：

BaseType_t xHigherPriorityTaskWoken = pdFALSE;
xSemaphoreGiveFromISR(xSem, &xHigherPriorityTaskWoken);
portYIELD_FROM_ISR(xHigherPriorityTaskWoken);

使用示例（资源池计数）：

// 最大 8 个资源块，初始全可用
SemaphoreHandle_t xPoolSem = xSemaphoreCreateCounting(8, 8);

if (xSemaphoreTake(xPoolSem, pdMS_TO_TICKS(10)) == pdTRUE)
{
   // 使用一个资源块
   // ...
   xSemaphoreGive(xPoolSem); // 归还
}

二值型信号量（Binary Semaphore）​

二值信号量只有两种状态：0（不可用）和 1（可用）。 最常见用途不是“资源计数”，而是“事件到没到”。

创建​

动态创建：

SemaphoreHandle_t xSemaphoreCreateBinary( void );
  • 返回值：成功返回句柄；失败返回 NULL。
  • 关键点：默认初始状态通常是“空”（0），需要先 give 一次才可 take 成功。

静态创建：

SemaphoreHandle_t xSemaphoreCreateBinaryStatic( StaticSemaphore_t *pxSemaphoreBuffer );

删除​

void vSemaphoreDelete( SemaphoreHandle_t xSemaphore );

删除语义和计数信号量一致。

Give / Take​

常用 API：

  • xSemaphoreTake
  • xSemaphoreGive
  • xSemaphoreGiveFromISR

典型 ISR 通知示例：

static SemaphoreHandle_t g_dmaDoneSem;

void vDmaConsumerTask(void *pv)
{
   for (;;)
   {
      if (xSemaphoreTake(g_dmaDoneSem, portMAX_DELAY) == pdTRUE)
      {
         // DMA 完成后处理数据
         ProcessDmaFrame();
      }
   }
}

void DMA_IRQHandler(void)
{
   BaseType_t xHigherPriorityTaskWoken = pdFALSE;

   // 清 DMA 中断标志...
   xSemaphoreGiveFromISR(g_dmaDoneSem, &xHigherPriorityTaskWoken);
   portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}

计数型 vs 二值型：如何选择​

对比项计数型信号量二值型信号量
计数范围0 ~ N0/1
主要语义多资源计数、事件累计单事件通知
多次 Give 是否累计会累计（到上限为止）通常不会累计（最多保持 1）
典型场景缓冲池、连接池、并发配额ISR 通知任务、简单任务同步
复杂度略高更轻量

选型速记：

  • 你在数“数量”：用计数型。
  • 你在等“发生没发生”：用二值型。

二值信号量 vs 互斥量（Mutex）​

二者都能 take/give，但设计目标不同，不能混用。

对比项二值信号量互斥量
目标事件同步共享资源互斥
所有权（owner）无严格 owner 语义有 owner，谁拿谁还
优先级继承不支持支持
ISR 中 Give支持（GiveFromISR）不支持
典型场景ISR -> 任务唤醒保护 UART/SPI/I2C/共享结构体

关键结论：

  • 防优先级反转的资源保护，必须用互斥量。
  • 中断唤醒任务，优先用二值信号量或任务通知。

信号量机制：任务如何阻塞与唤醒​

这一节结合 FreeRTOS 内核实现来描述信号量导致的阻塞与唤醒。

内核对象与关键数据结构​

  • 信号量本质上复用队列内核对象（Queue_t）。二值/计数信号量的 uxItemSize = 0，不搬运业务数据。
  • “令牌数量”由 uxMessagesWaiting 表示。
    • 二值信号量：取值通常为 0/1。
    • 计数信号量：取值范围 0 ~ uxLength（uxLength 即创建时最大计数）。
  • 阻塞任务通过事件链表管理：等待该信号量的任务挂在对象的 xTasksWaitingToReceive。
  • 同时等待超时的任务还会挂入延时链表（通过任务自身状态链表项），因此本质是“事件等待 + 时间等待”双条件阻塞。

xSemaphoreTake 的内核路径（任务态）​

xSemaphoreTake() 宏最终进入 xQueueSemaphoreTake()（queue.c）。核心过程：

  1. 进入临界区，检查 uxMessagesWaiting。
  2. 若 uxMessagesWaiting > 0：
    • 令牌数减一；
    • 若该对象属于队列集，会更新队列集可读事件；
    • 退出临界区并返回 pdTRUE。
  3. 若 uxMessagesWaiting == 0 且 xTicksToWait == 0：
    • 立即返回 errQUEUE_EMPTY（对信号量宏语义体现为 pdFALSE）。
  4. 若 uxMessagesWaiting == 0 且 xTicksToWait > 0：
    • 记录超时基准（vTaskSetTimeOutState）；
    • 挂入 xTasksWaitingToReceive（按优先级有序）；
    • 同步挂入延时链表，任务转为 Blocked；
    • 让出 CPU，等待 give 或超时。

xSemaphoreGive 的内核路径（任务态）​

xSemaphoreGive() 最终走队列发送路径（对信号量是“计数 +1”）：

  1. 进入临界区，检查是否可增加计数（是否到上限）。
  2. 可增加时将 uxMessagesWaiting 加一。
  3. 检查 xTasksWaitingToReceive：
    • 若非空，从事件链表移除一个最高优先级等待任务（同优先级按链表顺序）；
    • 被移除任务转回 Ready；
    • 若该任务优先级高于当前任务，请求一次上下文切换（抢占内核下通常立即生效）。
  4. 返回 pdTRUE；若计数已满则返回失败。

xSemaphoreGiveFromISR 的内核路径（中断态）​

  1. ISR 中调用 xSemaphoreGiveFromISR()，内核在中断上下文完成“计数 +1”和等待任务解除阻塞。
  2. 若唤醒了更高优先级任务，*pxHigherPriorityTaskWoken 被置为 pdTRUE。
  3. 退出 ISR 前调用 portYIELD_FROM_ISR(xHigherPriorityTaskWoken)：
    • 本质是请求 PendSV；
    • 真正的上下文切换在异常返回路径执行。

超时返回与竞争关系​

  • 任务在阻塞循环中通过 xTaskCheckForTimeOut() 判定是否超时。
  • 若超时先发生，任务从相关等待关系中移除并返回失败。
  • 若 give 先发生，任务被移入 Ready，最终是否立刻运行由调度器在就绪集合中统一裁决。

结论：信号量“唤醒”负责状态迁移（Blocked -> Ready），不直接等于“正在运行”；“运行决策”始终由调度器基于就绪优先级完成。

易错点清单（信号量）​

  • 把二值信号量长期当互斥量用：会失去优先级继承保护。
  • ISR 中误用任务态 API：必须改 FromISR 版本。
  • 创建二值信号量后从不 give 就直接 take：任务会长期阻塞。
  • take 超时后不判断返回值就继续访问资源：会引入隐蔽竞态。
  • 持有“受保护资源”期间做大量耗时操作：会放大系统延迟与抖动。
