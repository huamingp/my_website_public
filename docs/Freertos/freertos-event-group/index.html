事件组

事件组适合表达“多个条件位”的同步关系，比如：

  • 任务 A 等待 BIT0（DMA 完成）
  • 任务 B 等待 BIT1（传感器更新）
  • 任务 C 等待 BIT0 | BIT1（两个条件都满足后再运行）

它本质是一个“位标志集合 + 等待队列”。

概念与操作语义​

(1) 事件位与可用位数​

事件组类型是 EventBits_t，其底层位宽取决于 FreeRTOS 配置：

  • configUSE_16_BIT_TICKS = 1：EventBits_t 为 16 位，仅低 8 位可用于用户事件位。
  • configUSE_16_BIT_TICKS = 0：EventBits_t 为 32 位，仅低 24 位可用于用户事件位。

原因：最高 8 位是内核保留控制位，不给用户定义事件位。

纠正一个常见误解： “高 8 位表示与/或关系”不准确。与/或等待关系是由等待 API 的参数和内核控制标志共同决定，不是让用户直接把高 8 位当事件位使用。

(2) 与/或等待关系​

调用 xEventGroupWaitBits() 时可选择：

  • 等待任一位满足（OR 语义）
  • 等待所有位满足（AND 语义）

这由参数 xWaitForAllBits 决定：

  • pdFALSE：任一位满足即可唤醒
  • pdTRUE：所有目标位都满足才唤醒

(3) 唤醒与清位​

  • 事件组“置位”具备广播特性：一次置位可能唤醒多个等待任务。
  • “唤醒”不等于“立即运行”：最终谁先执行仍由调度器按优先级决定。
  • 等待任务可选择“退出等待时是否清位”：由 xClearOnExit 决定。

内部机制（结合源码执行链路）​

以下内容对应 FreeRTOS 内核文件 event_groups.c（不同版本细节名可能略有差异，但机制一致）。

(1) 核心数据结构​

事件组控制块（通常为 EventGroup_t / EventGroupDef_t）关键成员：

  • uxEventBits：当前事件位值（低位是用户事件位）。
  • xTasksWaitingForBits：等待该事件组的任务链表。

每个等待任务在链表项中携带“等待掩码 + 控制标志”，控制标志包括：

  • 是否等待全部位（ALL）
  • 唤醒时是否清位（CLEAR_ON_EXIT）

(2) 置位路径：xEventGroupSetBits​

核心流程可概括为：

  1. 进入临界区，将 uxBitsToSet OR 到 uxEventBits。
  2. 遍历 xTasksWaitingForBits 等待链表。
  3. 对每个等待任务计算“是否满足条件”：
    • OR 等待：(当前位 & 等待位) != 0
    • AND 等待：(当前位 & 等待位) == 等待位
  4. 满足则将该任务从等待链表移出并转入 Ready。
  5. 若该任务设置了“退出清位”，则把它请求清除的位累积到本次待清掩码。
  6. 遍历结束后统一清除应清位，再退出临界区并触发必要调度。

这解释了两个现象：

  • 一次置位可同时唤醒多个任务（广播）。
  • 清位不是“先到先清”，而是按内核规则统一处理，避免遍历中状态抖动。

(3) 等待路径：xEventGroupWaitBits​

核心流程：

  1. 读取当前 uxEventBits，先判断是否已满足条件。
  2. 若已满足：
    • 按 xClearOnExit 决定是否清位；
    • 立即返回（不阻塞）。
  3. 若不满足且 xTicksToWait = 0：立即超时返回。
  4. 若不满足且允许等待：
    • 任务把“等待位 + 控制标志”挂入 xTasksWaitingForBits；
    • 任务进入 Blocked，等待置位或超时。
  5. 任务被唤醒后返回“退出等待时刻的事件位快照”，供应用层判断是否因超时返回。

(4) ISR 路径：为什么很多 API 走 FromISR​

xEventGroupSetBitsFromISR() / xEventGroupClearBitsFromISR() 通常不会在 ISR 内直接完成全部链表遍历和任务状态迁移，而是通过“延后到守护任务（定时器服务任务）处理”的方式保证中断时延可控。

这也是为什么：

  • 需要使能定时器服务任务相关配置（configUSE_TIMERS 等）。
  • FromISR 版本返回值语义与任务态版本不同（重点是“是否成功提交请求”）。

API 函数（完整清单）​

以下为 FreeRTOS 常用事件组 API（以主线版本为准）。

创建与删除​

  1. 动态创建
EventGroupHandle_t xEventGroupCreate( void );
  • 返回值类型：EventGroupHandle_t
  • 返回值：成功返回事件组句柄，失败返回 NULL（常见原因：内存不足）
  1. 静态创建
EventGroupHandle_t xEventGroupCreateStatic( StaticEventGroup_t *pxEventGroupBuffer );
  • pxEventGroupBuffer：用户提供的事件组控制块内存地址
  • 返回值类型：EventGroupHandle_t
  • 返回值：成功返回句柄，失败返回 NULL
  1. 删除
void vEventGroupDelete( EventGroupHandle_t xEventGroup );
  • xEventGroup：要删除的事件组句柄
  • 返回值类型：void
  • 说明：动态创建对象会释放内核分配内存；静态创建对象仅使其失效，静态内存由用户管理

置位、清位、读位​

  1. 任务态置位
EventBits_t xEventGroupSetBits( EventGroupHandle_t xEventGroup,
                                const EventBits_t uxBitsToSet );
  • xEventGroup：目标事件组句柄
  • uxBitsToSet：要置 1 的事件位掩码（仅使用用户可用位）
  • 返回值类型：EventBits_t
  • 返回值：操作完成后的事件位值
  1. ISR 置位
BaseType_t xEventGroupSetBitsFromISR( EventGroupHandle_t xEventGroup,
                                      const EventBits_t uxBitsToSet,
                                      BaseType_t *pxHigherPriorityTaskWoken );
  • xEventGroup：目标事件组句柄
  • uxBitsToSet：要置位的掩码
  • pxHigherPriorityTaskWoken：若延后处理导致更高优先级任务应尽快运行，则置 pdTRUE
  • 返回值类型：BaseType_t
  • 返回值：pdPASS 表示请求提交成功，pdFALSE 表示失败（如延后处理队列满）
  1. 任务态清位
EventBits_t xEventGroupClearBits( EventGroupHandle_t xEventGroup,
                                  const EventBits_t uxBitsToClear );
  • xEventGroup：目标事件组句柄
  • uxBitsToClear：要清 0 的位掩码
  • 返回值类型：EventBits_t
  • 返回值：清位前的事件位值（便于调用者判断原状态）
  1. ISR 清位
BaseType_t xEventGroupClearBitsFromISR( EventGroupHandle_t xEventGroup,
                                        const EventBits_t uxBitsToClear );
  • xEventGroup：目标事件组句柄
  • uxBitsToClear：要清 0 的位掩码
  • 返回值类型：BaseType_t
  • 返回值：pdPASS 或 pdFALSE（语义与 FromISR 置位类似，表示请求是否成功提交）
  1. 任务态读取当前位
EventBits_t xEventGroupGetBits( EventGroupHandle_t xEventGroup );
  • xEventGroup：目标事件组句柄
  • 返回值类型：EventBits_t
  • 返回值：当前事件位值
  1. ISR 读取当前位
EventBits_t xEventGroupGetBitsFromISR( EventGroupHandle_t xEventGroup );
  • xEventGroup：目标事件组句柄
  • 返回值类型：EventBits_t
  • 返回值：当前事件位值（ISR 安全版本）

等待与同步​

  1. 等待事件位
EventBits_t xEventGroupWaitBits( EventGroupHandle_t xEventGroup,
                                 const EventBits_t uxBitsToWaitFor,
                                 const BaseType_t xClearOnExit,
                                 const BaseType_t xWaitForAllBits,
                                 TickType_t xTicksToWait );
  • xEventGroup：目标事件组句柄
  • uxBitsToWaitFor：等待位掩码（你关心哪些位）
  • xClearOnExit：pdTRUE 表示满足条件并退出时清除这些位；pdFALSE 表示不清
  • xWaitForAllBits：pdTRUE=AND（全部满足）；pdFALSE=OR（任一满足）
  • xTicksToWait：最大等待时间；0 表示不等待，portMAX_DELAY 表示可长期等待（取决于配置）
  • 返回值类型：EventBits_t
  • 返回值：函数返回时的事件位快照。需要结合 uxBitsToWaitFor 判断是“条件满足返回”还是“超时返回”
  1. 同步点（屏障）
EventBits_t xEventGroupSync( EventGroupHandle_t xEventGroup,
                             const EventBits_t uxBitsToSet,
                             const EventBits_t uxBitsToWaitFor,
                             TickType_t xTicksToWait );
  • xEventGroup：目标事件组句柄
  • uxBitsToSet：当前任务到达屏障时先置位的标志
  • uxBitsToWaitFor：屏障完成条件（通常是“所有参与者位都置位”）
  • xTicksToWait：最大等待时间
  • 返回值类型：EventBits_t
  • 返回值：同步点退出时的事件位快照

典型场景：3 个任务都准备好后一起进入下一阶段。每个任务先置自己的 bit，再等待“全体 bit 都到位”。

工程建议与易错点​

  1. 事件组是“位级同步”，不传业务数据。 需要传结构体/数据流时，仍应使用队列或消息缓冲。

  2. 用户只能使用低位事件位，禁止占用内核保留高 8 位。

  3. xEventGroupWaitBits() 返回后务必二次判定。 返回值是位快照，不是“布尔成功标志”，必须用位掩码判断是否真的满足条件。

  4. ISR 高频置位要关注定时器服务任务负载。 FromISR 版本通常是“投递请求”，如果投递队列打满会失败。

  5. 不要把事件组当互斥锁。 事件组没有 owner 和优先级继承语义，不能替代互斥量保护临界资源。
