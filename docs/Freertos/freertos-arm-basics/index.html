ARM相关基础

CMSIS-RTOS2​

osThreadAttr_t 是什么​

osThreadAttr_t 是 CMSIS-RTOS2 中用来描述线程创建属性的结构体类型。

名称拆解：

部分含义
osOperating System（操作系统）
Thread线程
AttrAttributes（属性）
_tC 语言类型后缀（typedef 定义的类型）

osThreadAttr_t = RTOS 线程创建时使用的属性配置结构体类型

常见字段：

字段名类型说明
nameconst char *线程名
priorityosPriority_t优先级枚举值
stack_memvoid *栈内存地址（NULL = RTOS 动态分配）
stack_sizeuint32_t栈大小（字节）
cb_memvoid *控制块内存地址（NULL = 动态分配）
cb_sizeuint32_t控制块内存大小（字节）
attr_bitsuint32_t属性标志位（如 osThreadJoinable）

不需要全部手动填写，未填的字段（0 / NULL）RTOS 会使用默认值。

线程启动参数​

pointer that is passed to the thread function as start argument

= 传递给线程入口函数的启动参数指针

处理器基础​

Word 大小​

Word = CPU 一次最自然处理的数据宽度（the natural data size of the processor）。Word 大小不固定，取决于处理器位宽：

CPU 位数Word 大小字节数
8 位8 bit1 byte
16 位16 bit2 byte
32 位32 bit4 byte
64 位64 bit8 byte

ARM 汇编速记​

参考：ARM 中常用汇编指令汇总

指令说明
MOV数据移动（寄存器间或立即数赋值）
LDR从内存读数据到寄存器
STR将寄存器数据写入内存
CMP比较（做减法但不保存结果，只更新标志位，配合条件跳转使用）
PUSH寄存器压栈，等效 STMDB SP!, {寄存器列表}
POP寄存器出栈，等效 LDMIA SP!, {寄存器列表}
B无条件跳转
B<cond>条件跳转，如 BEQ（相等跳转）、BNE（不等跳转）
BL带链接跳转，将返回地址存入 LR（R14），用于函数调用
BX跳转并可切换指令集状态；BX LR 是从函数返回的标准写法
MRS / MSR读/写特殊功能寄存器（如 BASEPRI、PRIMASK，FreeRTOS 关中断时常见）

栈与上下文切换​

栈增长方向​

ARM Cortex-M 使用 Full Descending 栈：

  • 栈向 低地址 增长
  • SP 始终指向 最后一次压入的数据（即栈顶，最低占用地址）

PUSH / POP 详解​

压栈示例：

PUSH {R0, R1, R2}   ; 等效于 STMDB SP!, {R0, R1, R2}

执行后内存布局（假设压栈前 SP = 1000）：

地址   内容
 996 │ R2 │  ← 旧 SP - 4  （较高地址）
 992 │ R1 │  ← 旧 SP - 8
 988 │ R0 │  ← 新 SP = 栈顶（最低地址）

规律：

  • 新 SP = 旧 SP − 4×N（N = 寄存器数量）
  • 编号 小 的寄存器存在 低地址（R0 在栈顶）
  • 编号 大 的寄存器存在 高地址（R2 离旧 SP 最近）

出栈示例：

POP {R0, R1, R2}    ; 等效于 LDMIA SP!, {R0, R1, R2}

按相同规则从栈中恢复，执行后 SP = 旧 SP + 12。

多寄存器 PUSH / POP 语法使用 {}，不是 []。

为什么函数调用要压栈​

  1. 保存返回地址：BL 把返回地址写入 LR（R14）。若被调函数内部还要调用其他函数，LR 会被覆盖，因此必须先 PUSH {LR} 保存，否则退出时无处可返。
  2. 保存执行现场：当前寄存器值、局部变量等。

什么是 "现场（Context）"​

程序在某一时刻的完整运行状态，包括：

  • 当前执行位置（PC 值）
  • 函数调用层级
  • 各寄存器的值
  • 栈中的局部变量与返回地址

任务切换本质​

任务切换 = 保存当前任务现场 + 恢复下一个任务现场

  • 入栈 = 保存当前执行状态
  • 出栈 = 恢复之前保存的执行状态

局部变量与 volatile​

情况存储位置
寄存器够用的普通局部变量优先放寄存器
地址被取（&a）或变量太大（数组等）强制放栈
加了 volatile通常放栈

volatile 的语义：告诉编译器 每次访问必须直接读写内存，不得缓存到寄存器。对局部变量加 volatile 后，编译器无法再把它留在寄存器里，会在栈上为其分配地址，从而保证每次都真实读写内存。

FreeRTOS 任务切换完整流程（Cortex-M）​

任务切换的本质是 任务上下文的保存与恢复。

触发： PendSV 异常（FreeRTOS 将其设为最低优先级异常，专门用于任务切换）

为什么 PendSV 设最低优先级？
确保任务切换发生在所有其他中断处理完毕之后。若在高优先级 ISR 执行期间发生切换，会导致上下文混乱。将 PendSV 设为最低，硬件会等到中断嵌套全部退出后才真正执行切换。

① 保存当前任务现场

步骤内容
硬件自动压栈xPSR, PC, LR, R12, R3–R0（异常入栈帧，顺序固定）
软件（PendSV Handler）压栈R4–R11

硬件异常入栈帧顺序（高地址 → 低地址）：
xPSR → PC → LR → R12 → R3 → R2 → R1 → R0
出栈时硬件按逆序自动恢复，最后弹出 PC 触发跳转，无需软件干预。 | 保存 SP | 将当前 PSP 写入当前任务 TCB 的 pxTopOfStack 字段 |

说明：每个任务拥有独立的栈空间。任务运行期间调用的所有函数，其局部变量和返回地址都压在 同一个任务栈 上（嵌套帧），并非各自独立的栈。

② 调度器选择下一个任务

根据优先级找出最高优先级的就绪任务，更新 pxCurrentTCB 指针。

③ 恢复下一个任务现场

步骤内容
取 SP从新任务 TCB 的 pxTopOfStack 字段读出 SP
软件出栈POP R4–R11
硬件自动出栈异常返回时自动恢复 R0–R3, R12, LR, PC, xPSR，最后跳转到 PC

④ CPU 继续执行新任务

为什么每个任务都要有自己的栈​

  • 各任务调用的函数不同 → 函数调用链不同
  • 各任务的局部变量不同
  • 需要独立保存/恢复各自的现场（上下文）
