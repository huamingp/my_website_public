同步与互斥

概念与目标​

在 FreeRTOS 中，同步与互斥都用于处理并发任务，但目标不同：

  • 同步（Synchronization）：协调“先后关系”，确保任务按预期时序执行。
  • 互斥（Mutual Exclusion）：保护“共享资源”，确保同一时刻只有一个执行体访问临界区。

典型示例：

  • 同步：任务 A 采样完成后，任务 B 才能处理数据。
  • 互斥：多个任务共用 UART/LCD/I2C 总线或同一全局数据结构时，必须串行访问。

一句话区分：同步关注“谁先谁后”，互斥关注“同一时刻只能有一个”。

常用同步机制与适用场景​

机制主要用途数据承载是否可用于 ISR 通知任务典型 API
队列（Queue）任务间传递数据 + 同步可以（拷贝数据项）可以（FromISR 版本）xQueueSend / xQueueReceive
二值信号量（Binary Semaphore）事件通知（有/无）不承载业务数据可以（xSemaphoreGiveFromISR）xSemaphoreTake / xSemaphoreGive
计数信号量（Counting Semaphore）资源计数、事件累计不承载业务数据可以（FromISR）xSemaphoreTake / xSemaphoreGive
事件组（Event Group）多条件按位同步位标志（bitmask）可在 ISR 置位（xEventGroupSetBitsFromISR）xEventGroupWaitBits
任务通知（Task Notification）轻量一对一通知可携带 32-bit 值可以（vTaskNotifyGiveFromISR 等）ulTaskNotifyTake / xTaskNotifyWait

说明：若是“一对一且追求低开销通知”，优先考虑任务通知；若要携带结构化数据，优先考虑队列。

互斥量与二值信号量的区别​

二者底层都基于队列机制，但语义与约束不同，不能混用。

对比项互斥量（Mutex）二值信号量（Binary Semaphore）
设计目的保护共享资源（互斥）任务/中断间事件同步
所有权（Owner）有。谁拿到谁释放，必须同一任务 Take/Give 成对无。任意上下文可 Give，接收方 Take
优先级继承支持（防止优先级反转）不支持
ISR 中使用不能在 ISR 中 Take/Give 互斥量可使用 FromISR 接口 Give
典型场景保护 UART、SPI、文件系统、共享链表ADC 完成中断通知任务、DMA 完成唤醒任务

关键纠错：

  • “二值信号量可以当互斥量用”在工程上不推荐。因为它没有优先级继承，容易在高低优先级任务竞争资源时出现优先级反转。
  • “互斥量可在中断里释放”是错误的。中断中不能使用互斥量接口，应改用二值信号量或任务通知进行 ISR 到任务的同步。

典型代码模式与 ISR 约束​

(1) 互斥量保护共享外设（任务与任务）​

static SemaphoreHandle_t g_uartMutex;

void UartTaskA(void *arg)
{
   for (;;)
   {
      if (xSemaphoreTake(g_uartMutex, pdMS_TO_TICKS(50)) == pdTRUE)
      {
         // 独占访问 UART
         Uart_SendString("A\r\n");
         xSemaphoreGive(g_uartMutex);
      }
      vTaskDelay(pdMS_TO_TICKS(20));
   }
}

要点：互斥量用于“保护资源”，而不是“通知事件”。

(2) 二值信号量实现 ISR 到任务通知​

static SemaphoreHandle_t g_adcDoneSem;

void AdcTask(void *arg)
{
   for (;;)
   {
      if (xSemaphoreTake(g_adcDoneSem, portMAX_DELAY) == pdTRUE)
      {
         // 收到 ADC 完成事件后再处理数据
         ProcessAdcData();
      }
   }
}

void ADC_IRQHandler(void)
{
   BaseType_t xHigherPriorityTaskWoken = pdFALSE;

   // 清中断标志...
   xSemaphoreGiveFromISR(g_adcDoneSem, &xHigherPriorityTaskWoken);
   portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}

要点：ISR 中必须使用 FromISR 版本 API，并按需触发 portYIELD_FROM_ISR。

(3) 创建接口与初始状态提醒​

  • 互斥量：xSemaphoreCreateMutex()
  • 二值信号量：xSemaphoreCreateBinary()

注意：xSemaphoreCreateBinary() 创建后默认是“空”的（不可立即 Take），通常需要先由事件源 Give 一次，或由 ISR 首次 Give 后再被任务 Take。

选型速记与常见误区​

选型速记：

  • 需要传数据：用队列。
  • 需要互斥访问共享资源：用互斥量。
  • 需要中断通知任务：用二值信号量或任务通知。
  • 需要统计 N 个可用资源：用计数信号量。
  • 需要等待多个条件组合：用事件组。

常见误区：

  • 用全局变量 + volatile 代替同步机制。volatile 只保证每次访问内存，不保证原子性和时序。
  • 长时间持有互斥量（在临界区做耗时计算或阻塞等待），会降低并发度并放大优先级反转影响。
  • 在拿着互斥量时再等待另一个互斥量，容易形成死锁。应统一加锁顺序并缩短临界区。

总结：同步用于“正确顺序”，互斥用于“安全访问”。在 FreeRTOS 实践中，先明确需求是“传数据、发事件还是锁资源”，再选择对应机制。

裸机“有缺陷互斥锁”与 FreeRTOS 对比​

这一小节用于说明：在裸机中可以用“全局变量 + volatile + 关中断”做一个最小可用互斥方案，但它并不是工程上完备的锁模型。

(1) 裸机最小互斥示例（有缺陷但可运行）​

static volatile uint8_t g_lock = 0;

static int lock_try_acquire(void)
{
   int ok = 0;

   __disable_irq();
   if (g_lock == 0)
   {
      g_lock = 1;
      ok = 1;
   }
   __enable_irq();

   return ok;
}

static void lock_release(void)
{
   __disable_irq();
   g_lock = 0;
   __enable_irq();
}

void task_like_loop(void)
{
   for (;;)
   {
      while (!lock_try_acquire())
      {
         // 忙等：抢不到锁就持续空转
      }

      // 临界区：访问共享资源
      // ...

      lock_release();
   }
}

说明：

  • volatile 仅保证每次都访问内存，不提供互斥语义。
  • 真正的“原子化”来自关中断保护 check-and-set 片段。
  • 抢锁失败后的忙等会持续占用 CPU。

(2) 该方案的核心缺陷​

维度裸机全局变量锁（关中断）的问题
CPU 利用率抢锁失败通常忙等，处理器空转，不能做其他有效工作
实时性临界区期间屏蔽中断，中断响应延迟增大，抖动变差
优先级管理无优先级继承，无法系统性处理优先级反转
可维护性易漏解锁、异常路径难处理，多资源时易死锁
可扩展性资源数量和并发路径增加后，复杂度快速上升

结论：缺陷不只在“效率低”，更关键是实时性和可维护性不可控。

(3) 反衬 FreeRTOS 的优势​

对照点裸机全局变量锁FreeRTOS 互斥量/同步机制
等待方式忙等占 CPU阻塞等待，CPU 可调度给其他任务
优先级反转基本无内建对策互斥量支持优先级继承
接口语义约定式、易误用API 分层清晰（任务上下文/ISR 上下文）
工程稳定性靠开发者自律内核提供统一机制和约束
