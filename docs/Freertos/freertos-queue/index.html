队列

队列（Queue）是 FreeRTOS 最常用的任务间通信机制之一，特点是：

  • 按 FIFO 传递数据项（也可配置从前端插入）
  • 内核负责并发访问保护
  • 可用于“传数据 + 同步事件”

API 详解​

创建队列​

动态创建：

QueueHandle_t xQueueCreate( UBaseType_t uxQueueLength,
                            UBaseType_t uxItemSize );
  • uxQueueLength：队列最多可容纳的数据项数量。
  • uxItemSize：每个数据项的字节大小。
  • 返回值：成功返回句柄，失败返回 NULL。

静态创建：

QueueHandle_t xQueueCreateStatic( UBaseType_t uxQueueLength,
                                  UBaseType_t uxItemSize,
                                  uint8_t *pucQueueStorageBuffer,
                                  StaticQueue_t *pxQueueBuffer );
  • uxQueueLength：队列最多可容纳的数据项数量。
  • uxItemSize：每个数据项的字节大小。
  • pucQueueStorageBuffer：用户提供的数据存储区首地址，大小至少为 uxQueueLength * uxItemSize 字节。
  • pxQueueBuffer：用户提供的控制块对象地址（StaticQueue_t）。
  • 返回值：成功返回队列句柄，失败返回 NULL。
  • 说明：静态创建不使用 heap，适合不希望运行期动态分配的场景。

写队列（Send）​

常用写入 API：

  • xQueueSend()：等价于发送到队尾。
  • xQueueSendToBack()：显式发送到队尾。
  • xQueueSendToFront()：发送到队头。
BaseType_t xQueueSend( QueueHandle_t xQueue,
                       const void *pvItemToQueue,
                       TickType_t xTicksToWait );
  • xQueue：目标队列句柄。
  • pvItemToQueue：待写入数据项地址，内核按 uxItemSize 拷贝入队。
  • xTicksToWait：等待时长。为 0 时不等待；大于 0 时可阻塞等待空位。
  • 队列满时：
    • xTicksToWait = 0：立即返回失败。
    • xTicksToWait > 0：任务进入 Blocked 等待空间。
  • 返回值（BaseType_t）：成功 pdPASS，失败 errQUEUE_FULL（宏值通常为 0）。

常用等价接口：

BaseType_t xQueueSendToBack( QueueHandle_t xQueue,
                                           const void *pvItemToQueue,
                                           TickType_t xTicksToWait );

BaseType_t xQueueSendToFront( QueueHandle_t xQueue,
                                             const void *pvItemToQueue,
                                             TickType_t xTicksToWait );
  • 参数与返回值语义与 xQueueSend 一致，仅入队位置不同。

ISR 对应写入 API：

BaseType_t xQueueSendFromISR( QueueHandle_t xQueue,
                              const void *pvItemToQueue,
                              BaseType_t *pxHigherPriorityTaskWoken );

BaseType_t xQueueSendToBackFromISR( QueueHandle_t xQueue,
                                    const void *pvItemToQueue,
                                    BaseType_t *pxHigherPriorityTaskWoken );

BaseType_t xQueueSendToFrontFromISR( QueueHandle_t xQueue,
                                     const void *pvItemToQueue,
                                     BaseType_t *pxHigherPriorityTaskWoken );
  • xQueue：目标队列句柄。
  • pvItemToQueue：待写入数据项地址（按 uxItemSize 拷贝）。
  • pxHigherPriorityTaskWoken：输出参数。若本次操作唤醒了更高优先级任务，会被置为 pdTRUE。
  • 返回值（BaseType_t）：成功 pdPASS；失败 errQUEUE_FULL。
  • 说明：ISR 版本不能阻塞（无 xTicksToWait）。退出中断前应调用 portYIELD_FROM_ISR(xHigherPriorityTaskWoken)。

读队列（Receive）​

BaseType_t xQueueReceive( QueueHandle_t xQueue,
                          void *pvBuffer,
                          TickType_t xTicksToWait );
  • xQueue：目标队列句柄。
  • pvBuffer：接收缓冲区地址，至少能容纳一个数据项（uxItemSize 字节）。
  • xTicksToWait：等待时长。为 0 时不等待，立即返回；大于 0 时可阻塞等待。
  • 返回值（BaseType_t）：成功 pdPASS；失败 pdFAIL（常见原因是超时且队列仍为空）。
  • 成功读取后，该数据项会从队列中移除。

ISR 对应读取 API：

BaseType_t xQueueReceiveFromISR( QueueHandle_t xQueue,
                                 void *pvBuffer,
                                 BaseType_t *pxHigherPriorityTaskWoken );
  • xQueue：源队列句柄。
  • pvBuffer：接收缓冲区地址，用于拷贝出一个数据项。
  • pxHigherPriorityTaskWoken：若本次接收让更高优先级任务转为 Ready，则置 pdTRUE。
  • 返回值（BaseType_t）：成功 pdPASS；失败 pdFAIL（队列空）。
  • 说明：ISR 版本不能阻塞；如 pxHigherPriorityTaskWoken == pdTRUE，退出中断前调用 portYIELD_FROM_ISR()。

查询与状态判断​

uxQueueMessagesWaiting

UBaseType_t uxQueueMessagesWaiting( const QueueHandle_t xQueue );
  • xQueue：目标队列句柄。
  • 返回值（UBaseType_t）：当前队列中已有的数据项个数。

uxQueueSpacesAvailable

UBaseType_t uxQueueSpacesAvailable( const QueueHandle_t xQueue );
  • xQueue：目标队列句柄。
  • 返回值（UBaseType_t）：当前剩余可写槽位个数。

xQueueIsQueueEmptyFromISR

BaseType_t xQueueIsQueueEmptyFromISR( const QueueHandle_t xQueue );
  • xQueue：目标队列句柄。
  • 返回值（BaseType_t）：空返回 pdTRUE，非空返回 pdFALSE。

xQueueIsQueueFullFromISR

BaseType_t xQueueIsQueueFullFromISR( const QueueHandle_t xQueue );
  • xQueue：目标队列句柄。
  • 返回值（BaseType_t）：满返回 pdTRUE，未满返回 pdFALSE。

查询函数适合监控和诊断，不适合替代同步逻辑。

覆盖（Overwrite）与偷看（Peek）​

xQueueOverwrite

BaseType_t xQueueOverwrite( QueueHandle_t xQueue,
                            const void *pvItemToQueue );
  • xQueue：目标队列句柄。
  • pvItemToQueue：待写入数据项地址。
  • 返回值（BaseType_t）：通常返回 pdPASS。
  • 限制：仅用于长度为 1 的队列（邮箱模型）。已有旧值会被新值覆盖。

xQueuePeek

BaseType_t xQueuePeek( QueueHandle_t xQueue,
                       void *pvBuffer,
                       TickType_t xTicksToWait );
  • xQueue：目标队列句柄。
  • pvBuffer：接收缓冲区地址。
  • xTicksToWait：等待时长。为 0 时不等待。
  • 返回值（BaseType_t）：成功 pdPASS；失败 pdFAIL。
  • 行为：读取队头但不移除数据项。

xQueueOverwriteFromISR

BaseType_t xQueueOverwriteFromISR( QueueHandle_t xQueue,
                                   const void *pvItemToQueue,
                                   BaseType_t *pxHigherPriorityTaskWoken );
  • 参数含义与任务态版本一致，额外的 pxHigherPriorityTaskWoken 用于请求退出中断后尽快切换。
  • 返回值（BaseType_t）：通常为 pdPASS。
  • 说明：仅用于长度为 1 的队列。

xQueuePeekFromISR

BaseType_t xQueuePeekFromISR( QueueHandle_t xQueue,
                              void * const pvBuffer );
  • xQueue：目标队列句柄。
  • pvBuffer：接收缓冲区地址。
  • 返回值（BaseType_t）：成功 pdPASS；失败 pdFAIL（队列空）。

队列集（Queue Set）​

概念​

队列集可以理解为“读一个集合，返回哪个成员有事件”。

  • 你不是直接从队列集拿业务数据。
  • 你先从队列集拿到“就绪成员句柄”。
  • 再根据这个句柄，去读对应的队列或取对应的信号量。

创建队列集​

QueueSetHandle_t xQueueCreateSet( const UBaseType_t uxEventQueueLength );
  • uxEventQueueLength：队列集内部“事件队列”的容量，不是字节数，而是可缓存的“事件令牌个数”。
  • 返回值（QueueSetHandle_t）：成功返回队列集句柄；失败返回 NULL。

容量概念：当某个成员对象出现一次“可读事件”（例如队列新入队 1 项，或信号量计数 +1），队列集内部会记录一个事件令牌。uxEventQueueLength 决定了最多可暂存多少个待处理事件。

成员管理 API​

BaseType_t xQueueAddToSet( QueueSetMemberHandle_t xQueueOrSemaphore,
                           QueueSetHandle_t xQueueSet );

BaseType_t xQueueRemoveFromSet( QueueSetMemberHandle_t xQueueOrSemaphore,
                                QueueSetHandle_t xQueueSet );
  • xQueueOrSemaphore：成员句柄（队列/二值信号量/计数信号量）。
  • xQueueSet：目标队列集句柄。
  • 返回值（BaseType_t）：pdPASS 成功；pdFAIL 失败。
  • 约束：成员在加入前应为空态或已对齐状态。不要把“已经堆积旧数据”的成员直接加入后马上 Select。

读队列集并联动读取成员​

QueueSetMemberHandle_t xQueueSelectFromSet( QueueSetHandle_t xQueueSet,
                                            TickType_t xTicksToWait );
  • xQueueSet：要等待的队列集。
  • xTicksToWait：等待超时（仅任务态版本）。
  • 返回值（QueueSetMemberHandle_t）：
    • 非 NULL：返回“就绪成员句柄”。
    • NULL：超时或无就绪成员。

联动规则如下：

  1. 先调用 xQueueSelectFromSet() 取回成员句柄。
  2. 用返回句柄和你已知的成员句柄逐个比较。
  3. 比较到哪个成员相等，就调用哪个成员对应的读取 API。

示例：

QueueSetMemberHandle_t m = xQueueSelectFromSet(g_set, portMAX_DELAY);

if (m == (QueueSetMemberHandle_t)g_rxQueue)
{
    uint8_t ch;
    (void)xQueueReceive(g_rxQueue, &ch, 0);
}
else if (m == (QueueSetMemberHandle_t)g_cmdQueue)
{
    Cmd_t cmd;
    (void)xQueueReceive(g_cmdQueue, &cmd, 0);
}
else if (m == (QueueSetMemberHandle_t)g_sem)
{
    (void)xSemaphoreTake(g_sem, 0);
}

队列集容量怎么估​

容量就是 uxEventQueueLength，它表示“队列集内部最多能暂存多少个事件通知”。

给出三档策略，按易用性从高到低：

  • 保守方案（推荐）： 设置为 所有加入队列集的队列长度之和 + 信号量个数。 理由：按最坏情况设计，即使多个队列同时堆满，也不容易丢事件通知。

  • 常规方案： 设置为 成员总数（队列数 + 信号量数）。 理由：适合大多数场景，默认任务被唤醒后会尽快处理就绪成员。 风险：如果同一个成员在短时间内连续产生多次事件，这个值可能偏小。

  • 最小方案： 设置为 1。 理由：仅适用于严格保证同一时间只有一个事件发生，且任务响应极快。 风险高，不推荐作为通用配置。

注意事项​

  • 一个“BUG”： 先运行任务实现写数据，写队列并写满后，再将该队列加入队列集 xQueueAddToSet()，随后读队列集xQueueSelectFromSet()尝试读队列数据，但是被xQueueSelectFromSet()一直阻塞。 现象是：队列明明有数据，xQueueSelectFromSet却不返回；生产者继续写时因为队列已满而写失败，系统就卡在“消费者一直等，生产者一直满”的状态。 根因不是“队列没数据”，而是“队列集事件令牌不是对历史状态补录”。队列加入队列集后，只有当后续发生写队列事件的时候才会生成令牌，才会成功读取队列集xQueueSelectFromSet。这就导致了队列在加入前就已经满了，后续又写不进去，同时该队列集的其他队列没有产生写队列事件，就不会产生新的事件令牌，于是cpu被xQueueSelectFromSet长期阻塞。 规避策略：
    1. 更换顺序（首选，最稳妥）：先加入队列集 xQueueAddToSet()，再启动生产者任务/中断数据流。 适用：系统初始化阶段、任务尚未放流时。 关键点：保证“成员进入集合时是空态”，后续每一次写入都能正常生成事件令牌。
    2. 在线切换修复（可用，但流程复杂，不建议作为常规方案）： 队列满 -> 暂停生产者（任务侧停发 + 中断侧屏蔽相关中断） -> 清空旧数据到一致状态 -> 加入集合 -> 恢复生产者。 关键点：
      • 暂停期间必须同时阻断“任务写入”和“ISR 写入”，否则清空后又被瞬间写满。
      • 清空要有边界条件（例如循环 xQueueReceive(..., 0) 直到失败），避免死循环。
      • 恢复后建议先观察一次消费者是否成功被唤醒，再放开高频数据源。
  • 不建议把互斥量放进队列集，会破坏互斥量优先级继承语义。
  • 队列集本身不是业务数据队列： 写数据时写成员队列（xQueueSend...）； 等事件时读队列集（xQueueSelectFromSet）； 拿到成员句柄后再读该成员（xQueueReceive 或 xSemaphoreTake）。 这三步缺一步都可能出现“看起来有事件但业务没读走”的假象。

数据分发：一对多（One-to-Many）​

一个生产者的数据需要被多个消费者“分别处理且互不影响”时，推荐使用“每个消费者一个私有队列”的广播模型。

  • 应用举例： 温湿度传感器每次采样后，数据需要同时给显示任务、算法分析任务、网络上报任务使用。

  • 推荐实现：队列数组 + 循环发送

    1. 为每个消费者创建独立队列（私有队列）。
    2. 生产者维护队列句柄数组。
    3. 生产者每次采样后，循环向每个队列发送一份副本。

    最小代码模板：

/* 核心思路：每个消费者一个私有队列，生产者循环广播 */
#define MAX_CONSUMER 2
static QueueHandle_t xConsumerQueues[MAX_CONSUMER];

void vSensorTask(void *params)
{
   SensorData_t data;
   for (;;)
   {
      data = ReadSensor();
      for (uint8_t i = 0; i < MAX_CONSUMER; i++)
      {
         xQueueSend(xConsumerQueues[i], &data, pdMS_TO_TICKS(2));
      }
      vTaskDelay(pdMS_TO_TICKS(20));
   }
}

// 消费者任务 A (分析)
void vAnalysisTask(void *pvParameters) {
    SensorData_t data;
    while(1) {
        // 只从自己的专属队列读
        if (xQueueReceive(xQ_Analysis, &data, portMAX_DELAY) == pdPASS) {
            Analyze_Temperature(data);
        }
    }
}
// 任务 B 和 C 同理，只是处理的函数不同

这样做的好处：

  • 消费者互不竞争同一个队列，不会出现“某个消费者读走后，其他消费者读不到”的问题。

  • 每个消费者可以按自己的节奏处理数据，互不拖慢。

  • 系统行为更可控，便于单独定位某个消费者的积压问题。

  • 发送策略（重点） 不建议在广播循环里使用 portMAX_DELAY 无期限阻塞，否则某个慢消费者队列满了会把生产者卡死。

    常见策略：

    1. 短超时发送（推荐默认）：xQueueSend(..., pdMS_TO_TICKS(1~5))
      • 好处：给消费者一点处理时间，但不会长期阻塞生产者。
    2. 非阻塞发送：xQueueSend(..., 0)
      • 适合高实时场景；若队列满可直接丢弃该消费者本次数据并计数告警。
    3. 最新值覆盖模型（仅长度为 1 的邮箱队列）
      • 使用 xQueueOverwrite() / xQueueOverwriteFromISR()，消费者总是读到“最新值”，不追求每条历史样本都保留。

  • 容量与节拍建议

    1. 先估算每个消费者最大处理周期 T_consumer_max 与生产周期 T_prod。
    2. 队列长度至少满足约束：队列长度 >= ceil(T_consumer_max / T_prod)。
    3. 工程上再留 20%~50% 裕量，防止突发抖动导致满队列。

  • ISR 场景补充 若生产者在中断里产出数据：

    • 使用 xQueueSendFromISR() 或邮箱模型的 xQueueOverwriteFromISR()。
    • 用 pxHigherPriorityTaskWoken + portYIELD_FROM_ISR() 及时唤醒消费者。

  • 易错点

    1. 误把“一个公共队列 + 多消费者”当广播：这是竞争消费，不是一对多广播。
    2. 广播循环里无限阻塞：一个慢消费者会拖垮所有消费者。
    3. 所有消费者共用同一丢包策略：应按任务重要性分别配置（例如显示可丢、控制不可丢）。

代码框架（最小模板）​

//TempHumiTask.c
/* 一对多广播：1个生产者 -> N个消费者（每个消费者1个私有队列） */
#include "FreeRTOS.h"
#include "task.h"
#include "queue.h"

#define MAX_CONSUMER 2
#define QUEUE_LEN  10

typedef struct
{
   float fTemperatureC;
   float fHumidityRH;
   uint32_t ulTick;
} SensorData_t;

uint8_t g_queue_cnt = 0;
QueueHandle_t  g_xQueues[MAX_CONSUMER];


/* 业务函数（示例） */
static SensorData_t xReadSensorOnce(void);
static void vDisplayTaskProcess(const SensorData_t *pxData);
static void vUploadTaskProcess(const SensorData_t *pxData);

/* 生产者：读取一次，分别投递到各私有队列 */
static void vProducerTask(void *pvParameters)
{
   SensorData_t xData;


   for (;;)
   {
      xData = xReadSensorOnce();

      for (uint8_t i = 0; i < MAX_CONSUMER; i++)
      {
      xQueueSend(g_xQueues[i], &xData, pdMS_TO_TICKS(2));    
      }
      vTaskDelay(pdMS_TO_TICKS(20));
   }
}

/* 消费者A：显示 */
static void vDisplayTask(void *pvParameters)
{
    // 从参数中还原队列句柄
    QueueHandle_t xQueue_Display = (QueueHandle_t)pvParameters;
    SensorData_t xData;
   for (;;)
   {
      if (xQueueReceive(xQueue_Display, &xData, portMAX_DELAY) == pdPASS)
      {
         vDisplayTaskProcess(&xData);
      }
   }
}

/* 消费者B：上报 */
static void vUploadTask(void *pvParameters)
{
   // 从参数中还原队列句柄
   QueueHandle_t xQueue_Upload = (QueueHandle_t)pvParameters;
   SensorData_t xData;
   for (;;)
   {
      if (xQueueReceive(xQueue_Upload, &xData, portMAX_DELAY) == pdPASS)
      {
         vUploadTaskProcess(&xData);
      }
   }
}

void RegisterQueueHandle(QueueHandle_t queueHandle)
{
	if (g_queue_cnt < MAX_CONSUMER)
	{
		g_xQueues[g_queue_cnt] = queueHandle;
		g_queue_cnt++;
	}
}
// main.c
/* 初始化：创建私有队列并启动任务 */
void vAppInit(void)
{
   //创建显示任务队列
   QueueHandle_t  xQueue_Display = xQueueCreate(QUEUE_LEN, sizeof(SensorData_t));
   //注册队列
   RegisterQueueHandle(xQueue_Display);
   //创建上报任务队列
   QueueHandle_t xQueue_Upload = xQueueCreate(QUEUE_LEN, sizeof(SensorData_t));
   //注册队列
   RegisterQueueHandle(xQueue_Upload);

   xTaskCreate(vProducerTask, "producer", 256, NULL, 3, NULL);
   xTaskCreate(vDisplayTask, "display", 256, (void*)xQueue_Display, 2, NULL);
   xTaskCreate(vUploadTask, "upload", 256, (void*)xQueue_Upload, 2, NULL);
}

框架说明：

  • 生产者不直接给多个任务共享同一个队列，而是“每个消费者一个队列”。
  • 广播循环使用短超时发送，避免单个慢消费者拖死全系统。
