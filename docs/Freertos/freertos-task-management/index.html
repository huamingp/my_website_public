任务管理

创建任务​

FreeRTOS 提供两种创建任务的方式：动态内存分配和静态内存分配。

动态创建：xTaskCreate​

BaseType_t xTaskCreate(
    TaskFunction_t pxTaskCode,               // 任务函数指针
    const char * const pcName,              // 任务名（仅调试用）
    const configSTACK_DEPTH_TYPE usStackDepth, // 栈大小，单位 word
    void * const pvParameters,              // 传入任务函数的参数
    UBaseType_t uxPriority,                 // 任务优先级
    TaskHandle_t * const pxCreatedTask      // 输出：任务句柄
);
参数说明
pxTaskCode任务函数指针。任务函数应永远不退出，或在退出时调用 vTaskDelete(NULL)。
pcName任务名称，仅用于调试，FreeRTOS 内部不使用。长度上限为 configMAX_TASK_NAME_LEN。
usStackDepth栈大小，单位 word（32 位 MCU 上 1 word = 4 字节）。传入 100 = 400 字节。
pvParameters调用任务函数时传入的参数：pxTaskCode(pvParameters)。
uxPriority优先级，范围 0 ~ (configMAX_PRIORITIES – 1)，数值越大优先级越高。传入过大值会被自动裁剪。
pxCreatedTask输出任务句柄，后续操作该任务（如修改优先级）时使用。不需要可传 NULL。
返回值成功返回 pdPASS；失败返回 errCOULD_NOT_ALLOCATE_REQUIRED_MEMORY（值为 -1，内存不足）。注意：并非 pdFAIL。

常见疑问：TaskFunction_t 与 pvParameters​

我的疑问：

  1. TaskFunction_t 是函数指针，为什么 xTaskCreate() 常直接写任务函数名？
  2. 为什么任务函数规范是 void ATaskFunction( void *pvParameters )，不能写成其他参数类型？

先看类型定义：

typedef void (*TaskFunction_t)( void * );

结论先记：

  • 在 C 语言里，函数名在大多数表达式中会转换为函数指针，所以这里传 MyTask 和 &MyTask 等价。
  • FreeRTOS 会按 TaskFunction_t 的固定原型调用任务入口函数。
  • pvParameters 设计成 void *，目的是让内核与参数具体类型解耦。

1) 为什么函数名可以直接传入

xTaskCreate 第一个参数要求的是函数指针，因此下面两种写法都可以：

xTaskCreate(MyTask,  "T1", 256, NULL, 2, NULL);
xTaskCreate(&MyTask, "T1", 256, NULL, 2, NULL);

日常工程一般省略 &，可读性更好。

2) 为什么必须是 void (*)(void *)

内核调用任务入口时，等价于执行 pxTaskCode(pvParameters)。 如果你写成 void MyTask(int x)、void MyTask(void) 或其他签名，就不是兼容的函数指针类型：

  • 编译期会出现不兼容告警（或被强转掩盖）。
  • 语言标准层面属于未定义行为（UB），结果不可依赖。

所以更准确的说法是：不是语法上“绝对不能写”，而是接口契约要求必须匹配该原型，工程上必须遵守。

3) 需要多个参数时怎么做

把参数打包进结构体，再通过 pvParameters 传递其指针：

typedef struct
{
   UART_HandleTypeDef *huart;
   uint8_t channel;
   uint16_t period_ms;
} TaskArg_t;

static TaskArg_t g_txArg = {
   .huart = &huart1,
   .channel = 2,
   .period_ms = 100,
};

void TxTask(void *pvParameters)
{
   TaskArg_t *arg = (TaskArg_t *)pvParameters;

   for (;;)
   {
      // 使用 arg 中的参数
      vTaskDelay(pdMS_TO_TICKS(arg->period_ms));
   }
}

xTaskCreate(TxTask, "Tx", 256, &g_txArg, 2, NULL);

注意：传入参数对象的生命周期必须覆盖任务运行期。不要把即将失效的栈上局部变量地址传给任务。

补充：如果任务不需要参数，pvParameters 传 NULL 即可。

静态创建：xTaskCreateStatic​

TaskHandle_t xTaskCreateStatic(
    TaskFunction_t pxTaskCode,           // 任务函数指针
    const char * const pcName,          // 任务名（仅调试用）
    const uint32_t ulStackDepth,        // 栈大小，单位 word
    void * const pvParameters,          // 传入任务函数的参数
    UBaseType_t uxPriority,             // 任务优先级
    StackType_t * const puxStackBuffer, // 用户提供的栈数组
    StaticTask_t * const pxTaskBuffer   // 用户提供的 TCB 结构体
);
参数说明
pxTaskCode同上
pcName同上
ulStackDepth栈大小，单位 word
pvParameters同上
uxPriority同上
puxStackBuffer用户提供的栈内存（数组），大小 = ulStackDepth × 4 字节
pxTaskBuffer用户提供的 StaticTask_t 结构体指针
返回值成功返回任务句柄；失败返回 NULL

动态 vs 静态创建对比​

对比项xTaskCreate（动态）xTaskCreateStatic（静态）
TCB & 栈来源FreeRTOS heap（pvPortMalloc）用户提供的静态缓冲区
是否依赖 heap是否
创建失败可能可能（内存不足）不会（编译期确定）
删除任务后内存系统自动释放 TCB 和栈TCB 和栈不释放（用户管理）
内存碎片可能产生不产生
适用场景一般应用安全关键 / 无堆分配场景

合理估算任务栈大小​

任务栈需要容纳三部分内容：函数调用链 + 局部变量 + 上下文切换现场。

上下文切换现场（固定开销）​

每次任务切换，FreeRTOS 将完整寄存器现场保存到任务栈：

阶段保存内容字节数
硬件自动压栈（异常入栈帧）xPSR, PC, LR, R12, R3–R0（共 8 个）32 字节
软件（PendSV Handler）压栈R4–R11（共 8 个）32 字节
合计16 个寄存器64 字节

若使用 FPU，栈开销会增加。Cortex-M4F/M7F 在异常入栈时可能增加浮点扩展帧（常见为 72 字节，含保留字），若任务切换时还需软件保存高位浮点寄存器（s16–s31），额外开销还会继续增加。工程上应通过高水位线实测，而不是只按固定值估算。

函数调用栈帧（逐层叠加）​

每一层函数调用都会在栈上建立一个 栈帧，包含：

  • 返回地址（若该函数内部还有子调用，LR 需压栈保存）
  • 编译器选择保留的 Callee-saved 寄存器（R4–R11 中实际用到的部分，通常 4–6 个）
  • 该层函数的局部变量

注：不是每层都会保存全部 9 个寄存器（R4–R11 + LR），实际由编译器按需压栈，简单函数可能只压 2–4 个。

估算步骤​

  1. 确定 最深调用链 的层数（最坏情况下同时活跃的嵌套层数）
  2. 每层估计 32 字节（保守值：含 LR + 约 4 个寄存器 + 少量局部变量）
  3. 加上最深层函数中所有 局部变量 的总大小（注意数组、缓冲区）
  4. 加上上下文切换固定开销 64 字节
  5. 向上对齐到 8 字节，再预留 20%–30% 余量

经验值：

  • 简单任务（2–3 层调用，无大缓冲区）→ 128–256 字节（32–64 word）
  • 含 printf / sprintf 的任务 → 至少 512 字节（128 word），因格式化函数调用链深、内部缓冲区大

运行时验证方法​

方法说明
uxTaskGetStackHighWaterMark(handle)返回历史最小剩余栈空间（单位 word），越小越危险
configCHECK_FOR_STACK_OVERFLOW 1 或 2栈溢出时触发 vApplicationStackOverflowHook()，便于定位问题
开发阶段先给大栈再缩减先分配 512 word，稳定运行后根据水位线结果收紧

任务中优先使用局部变量​

多个任务共享同一个全局变量是不安全的。

根本原因：非原子操作 + 任意时刻均可被抢占​

看似一行的 C 代码，编译后往往是 多条指令：

counter++;
// 实际编译为：
// LDR  R0, [&counter]   ; 从内存读
// ADD  R0, R0, #1       ; 加一
// STR  R0, [&counter]   ; 写回内存

任务切换可以发生在这三条指令的任意间隙。若任务 A 和 B 同时执行 counter++，初始值为 5：

任务A: LDR R0 = 5          ← 读到 5
--- 切换到任务B ---
任务B: LDR R0 = 5          ← 也读到 5（任务A 还没写回）
任务B: ADD R0 = 6
任务B: STR → counter = 6
--- 切换回任务A ---
任务A: ADD R0 = 6          ← A 的寄存器快照仍是旧值 5
任务A: STR → counter = 6   ← 覆盖了任务B 的结果！

最终 counter = 6，而非预期的 7。这就是典型的 竞态条件（Race Condition），导致 数据丢失。

常用保护方式​

方式适用场景
taskENTER_CRITICAL() / taskEXIT_CRITICAL()临界区短、读写简单
互斥量 Mutex（xSemaphoreCreateMutex）保护较复杂的共享资源
队列 Queue（xQueueSend / xQueueReceive）任务间传递数据，天然线程安全
挂起调度器（vTaskSuspendAll）不涉及中断的短临界区

创建顺序 ≠ 执行顺序​

即使优先级相同，任务的创建顺序和启动后的执行顺序也不一定一致。

原因​

  1. 调度器未启动前任务均处于就绪态：xTaskCreate 只是把任务放入就绪队列，真正执行要等 vTaskStartScheduler() 调用后才开始。

  2. vTaskStartScheduler() 内部还会额外创建任务：Idle Task（必须）以及 Timer Daemon Task（若启用 configUSE_TIMERS），这些任务也会插入就绪队列，改变队列的头尾顺序。

  3. 优先级是第一决定因素：高优先级任务无论何时被创建，总是优先执行，与创建顺序无关。

  4. 同优先级的轮转顺序会动态变化：任务因延时、阻塞等原因被移出就绪队列后重新加入时，会追加到队列 尾部，导致后续轮转顺序与初始创建顺序不同。

结论： 绝不要依赖创建顺序来控制执行时序。需要严格排序时，应使用 优先级差异 或 信号量 / 事件标志 进行显式同步。

任务删除：vTaskDelete​

void vTaskDelete( TaskHandle_t xTaskToDelete );
参数说明
xTaskToDelete目标任务的句柄（由 xTaskCreate 返回）；传入 NULL 表示删除 自身

注意事项​

  • 内存延迟回收：调用后任务的 TCB 和任务栈 不会立即释放，需等 Idle Task 运行时 才真正回收（仅限动态创建的任务）。若系统负载高、Idle Task 长时间得不到执行，内存将无法及时释放。
  • 静态任务不受影响：静态创建的任务（xTaskCreateStatic）的 TCB 和栈由用户管理，vTaskDelete 不会释放它们。
  • 内存碎片风险：频繁动态创建/删除任务会使 heap 碎片化，长期运行后可能导致分配失败。
  • RTOS 设计哲学：任务不推荐频繁删除。任务通常是长期存在的功能模块：
    • 需要 暂停 时 → 使用 vTaskSuspend() 或阻塞队列/信号量
    • 需要 减少 CPU 占用 时 → 降低优先级或进入阻塞等待
    • 确实需要终止 时（如检测到严重异常）→ 才考虑 vTaskDelete

Tick 时钟基准：为何不用 SysTick 给 HAL​

结论：FreeRTOS 本身占用 SysTick 作为调度 Tick；HAL 库也需要一个独立的时基，两者共用同一个 SysTick 会产生冲突，因此建议将 HAL 的时基改为普通定时器（如 TIM6）。

背景：两个 "用户" 争用同一个 SysTick​

模块对 SysTick 的用途
FreeRTOS作为调度 Tick（configTICK_RATE_HZ，通常 1000 Hz），每个 SysTick 中断调用 xPortSysTickHandler() 推进任务调度
STM32 HAL作为 uwTick 计数源，驱动 HAL_GetTick() / HAL_Delay() 及所有外设驱动的超时计算

当 FreeRTOS 启用后，它接管 SysTick 中断向量，HAL 里的 HAL_IncTick() 不再被默认调用，uwTick 停止更新 → HAL_Delay() 永久阻塞，外设超时判断失效。

具体冲突原因​

  1. 中断优先级约束
    FreeRTOS 要求所有调用 FreeRTOS API 的中断优先级必须 ≤ configMAX_SYSCALL_INTERRUPT_PRIORITY（数值上 ≥ 该值）。SysTick 被 FreeRTOS 配置为特定优先级。但 HAL 有时需要在更高优先级 ISR 里调用 HAL_GetTick() 判断超时，若共用 SysTick 则优先级约束冲突。

  2. 调度器挂起期间 HAL 超时失效
    FreeRTOS 在临界区（taskENTER_CRITICAL）或挂起调度器（vTaskSuspendAll）期间会屏蔽低于阈值的中断，SysTick 中断被屏蔽 → uwTick 不再递增 → 所有 HAL 超时计算停摆，导致 HAL 驱动挂死。

  3. HAL_Delay() 本身不适合在任务中使用
    HAL_Delay() 是轮询 uwTick 的忙等循环，在 RTOS 任务里不会主动进入阻塞态，虽然中断和调度仍可发生，但会无谓占用 CPU 时间并影响同优先级任务的实时性。任务内应统一改用 vTaskDelay() / osDelay()。

推荐做法（STM32CubeMX 标准配置）​

SysTick  ──►  FreeRTOS 独占（调度 Tick，1 ms）
TIM6     ──►  HAL timebase（HAL_Delay / HAL_GetTick）

在 CubeMX 中：Pinout & Configuration → SYS → Timebase Source 选择任意空闲 TIMx（推荐无 PWM 输出的基本定时器，如 TIM6 / TIM7）。

生成代码后，HAL_InitTick() 用 TIMx 替代 SysTick，HAL_IncTick() 在 TIMx 中断里调用，两套时基完全解耦。

总结对比​

对比项共用 SysTick（不推荐）SysTick(FreeRTOS) + TIMx(HAL)（推荐）
HAL_Delay() 可靠性调度器挂起时失效始终正常
中断优先级冲突存在无
uwTick 更新可能被中断屏蔽打断始终连续
硬件资源消耗少一个定时器多占一个 TIMx

四种任务状态​

FreeRTOS 中任务常见运行状态如下。

状态含义典型进入方式
Running当前正在 CPU 上执行调度器选中该任务
Ready已满足运行条件，等待被调度创建完成、阻塞解除、被恢复
Blocked等待时间到期或等待事件（队列/信号量/通知）vTaskDelay、xQueueReceive、xSemaphoreTake 等
Suspended被显式挂起，不参与常规就绪调度vTaskSuspend

状态转换关系​

  • Running -> Blocked：任务主动等待时间或事件。
  • Blocked -> Ready：等待条件满足（超时到达、收到消息、拿到信号量）。
  • Running -> Ready：被更高优先级任务抢占，或同优先级时间片轮转。
  • Ready -> Running：被调度器选中获得 CPU。
  • Running/Ready/Blocked -> Suspended：被 vTaskSuspend 挂起。
  • Suspended -> Ready：被 vTaskResume 或 xTaskResumeFromISR 恢复。

相关函数​

场景常用函数
时间阻塞vTaskDelay、vTaskDelayUntil
事件阻塞xQueueReceive、xSemaphoreTake、ulTaskNotifyTake
挂起任务vTaskSuspend
恢复任务vTaskResume、xTaskResumeFromISR

下面按“函数做什么 + 什么时候醒来 + 适用场景”逐个说明。

(1) vTaskDelay(ticks)​

  • 作用：让当前任务延时一段 Tick 时间。
  • 状态变化：Running -> Blocked，到时后自动变 Ready。
  • 参数含义：ticks 是 Tick 数，不是毫秒。通常配合 pdMS_TO_TICKS(ms) 使用。
  • 典型用途：周期性执行但对周期误差不敏感的任务。
vTaskDelay(pdMS_TO_TICKS(100)); // 当前任务延时约 100ms

(2) vTaskDelayUntil(&lastWakeTime, periodTicks)​

  • 作用：按固定节拍运行任务，减少周期抖动。
  • 状态变化：Running -> Blocked，到下一个绝对唤醒点后转 Ready。
  • 参数含义：
    • lastWakeTime：上次唤醒时刻（需要先初始化为 xTaskGetTickCount()）。
    • periodTicks：周期 Tick。
  • 典型用途：控制环、采样任务等“严格周期任务”。
TickType_t last = xTaskGetTickCount();
for (;;)
{
      // 周期任务逻辑
      vTaskDelayUntil(&last, pdMS_TO_TICKS(10)); // 10ms 固定节拍
}

(3) xQueueReceive(queue, &item, timeout)​

  • 作用：从队列读一条消息。
  • 状态变化：
    • 队列有数据：立即返回，任务通常保持 Running。
    • 队列为空且 timeout > 0：任务进入 Blocked 等待数据或超时。
  • 返回值：pdPASS 表示收到数据；超时返回失败值。
  • 典型用途：任务间数据传递（生产者-消费者模型）。

(4) xSemaphoreTake(sem, timeout)​

  • 作用：尝试获取信号量/互斥量。
  • 状态变化：拿不到且设置等待时间时进入 Blocked，拿到后继续运行。
  • 返回值：成功 pdTRUE，超时 pdFALSE。
  • 典型用途：
    • 二值信号量：事件同步。
    • 互斥量：共享资源互斥访问（带优先级继承）。

(5) ulTaskNotifyTake(clearOnExit, timeout)​

  • 作用：等待“任务通知”计数值，大多数场景可替代二值信号量。
  • 状态变化：通知值为 0 且需要等待时进入 Blocked。
  • 返回值：返回取到的通知计数值（0 表示超时）。
  • 典型用途：ISR 与任务、任务与任务之间的轻量同步，开销比队列更低。

(6) vTaskSuspend(taskHandle)​

  • 作用：把任务挂起，不再参与常规调度。
  • 状态变化：目标任务进入 Suspended。
  • 特点：不会因为时间到或事件到而自动恢复。
  • 典型用途：人为暂停某个功能任务，等待外部命令再恢复。

(7) vTaskResume(taskHandle) / xTaskResumeFromISR(taskHandle)​

  • 作用：恢复被挂起任务。
  • 状态变化：Suspended -> Ready，随后是否立刻运行取决于优先级。
  • 区别：
    • vTaskResume：在任务上下文中调用。
    • xTaskResumeFromISR：在中断中调用。

(8) 选型速记​

  • 需要“等固定时间”：优先用 vTaskDelayUntil。
  • 需要“收到数据再继续”：用 xQueueReceive。
  • 需要“保护共享资源”：用 xSemaphoreTake（互斥量）。
  • 需要“最快的事件唤醒”：优先考虑任务通知 ulTaskNotifyTake。
  • 需要“人工停用任务”：用 vTaskSuspend / vTaskResume。

说明：Suspended 与 Blocked 的关键区别在于，Blocked 可由事件或超时自动唤醒；Suspended 必须由其他上下文显式恢复。

任务管理与调度（内核视角）​

这一部分可以理解为两件事：

  • 管理：任务放在哪些列表里。
  • 调度：什么时候把 CPU 从当前任务切给另一个任务。

(1) 核心控制块与关键指针​

  • TCB（Task Control Block）：每个任务对应一个，保存任务栈顶指针、优先级、状态等。
  • pxCurrentTCB：始终指向“当前正在运行”的任务 TCB。
  • uxTopReadyPriority：当前就绪任务中的最高优先级索引，便于快速找到下一个任务。

(2) 任务管理依赖的主要列表​

  • 就绪列表（Ready List）：按优先级分组。每个优先级对应一个就绪链表。
  • 延时列表（Delayed List）：等待超时到期的任务。
  • 溢出延时列表（Overflow Delayed List）：处理 Tick 计数回卷时的延时任务。
  • 挂起列表（Suspended List）：被显式挂起、不会自动恢复的任务。
  • 等待事件列表（Event List）：等待队列/信号量/通知的任务。

关键理解：任务状态变化，本质上是任务节点在不同链表之间移动。

(3) Tick 中断主要做什么​

SysTick 周期到来后（或等效 Tick 源）通常会做三类事情：

  1. Tick 计数加一（内核时基推进）。
  2. 检查延时任务是否到期；到期任务从 Delayed List 移到 Ready List。
  3. 若出现更高优先级就绪任务，或启用时间片且同优先级需轮转，则请求一次上下文切换（触发 PendSV）。

(4) 高优先级先运行与同优先级轮转​

  • 高优先级抢占：只要有更高优先级任务进入 Ready，当前较低优先级任务会被抢占。
  • 同优先级轮转：当 configUSE_TIME_SLICING = 1 时，同优先级任务按时间片轮流获得 CPU。
  • 让出 CPU 的常见方式：
    • 主动阻塞（vTaskDelay、等待队列/信号量）
    • 主动让步（taskYIELD）
    • 被更高优先级任务抢占

(5) 一条完整的调度链路（示例）​

假设任务 A（低优先级）正在运行，任务 B（高优先级）在延时列表等待：

  1. Tick 到来，内核发现 B 延时到期。
  2. B 从 Delayed List 移入对应优先级的 Ready List。
  3. 内核比较优先级，发现 B 高于 A，发起调度请求。
  4. PendSV 执行上下文切换：保存 A 的现场，恢复 B 的现场。
  5. pxCurrentTCB 更新为 B，CPU 开始运行 B。

空闲任务与 Hook 函数​

空闲任务（Idle Task）由内核在启动调度器时自动创建，优先级固定为 tskIDLE_PRIORITY（通常为 0）。它的核心作用不是“做业务逻辑”，而是为系统提供最低优先级背景处理能力。

(1) 空闲任务的作用与运行特性​

  • 动态任务删除后的资源回收：被 vTaskDelete() 删除的动态任务，其 TCB 与栈由空闲任务在合适时机统一释放。
  • 系统保底运行任务：当没有其他就绪任务时，CPU 执行空闲任务，避免处理器“无任务可跑”。
  • 调度特性：空闲任务不会主动抢占用户任务；只要存在更高优先级就绪任务，空闲任务就不会运行。

关键结论：若高优先级任务长期不阻塞（例如 for(;;) 忙等），空闲任务可能长期得不到调度，导致已删除动态任务的内存回收不及时。

(2) Hook 函数是什么​

Hook 可以理解为“内核在固定时机预留给用户的扩展入口”。

  • 本质上，Hook 也是回调函数。
  • 区别在于：Hook 的触发时机和函数名由内核约定，用户只负责按接口实现；一般“回调函数”是更宽泛的概念，不一定由 RTOS 固定约束。

FreeRTOS 常见 Hook：

  • vApplicationIdleHook()：空闲任务循环中调用。
  • vApplicationTickHook()：每次 Tick 中断时调用。
  • vApplicationMallocFailedHook()：动态分配失败时调用。
  • vApplicationStackOverflowHook()：检测到任务栈溢出时调用。

(3) Hook 的使用场景与约束​

  • Idle Hook 适合做低频、可中断、非实时的后台工作。示例：在 STM32 电机小车项目中，当所有业务任务都阻塞时，在 vApplicationIdleHook() 里读取一次 uxTaskGetStackHighWaterMark() 并统计最小剩余栈水位，结果写入全局诊断结构体，供串口调试任务下一次唤醒后统一打印。
  • Tick Hook 适合做非常短小且高频的时间基准处理。示例：系统 Tick 为 1ms 时，在 vApplicationTickHook() 中仅执行 softTimerCnt++，再由普通任务每 10ms 检查该计数并处理按键消抖或软件定时器到期，避免在中断里直接做复杂逻辑。
  • Malloc/StackOverflow Hook 适合故障记录、告警上报或触发安全降级。示例：vApplicationMallocFailedHook() 中立刻关闭非关键任务创建并点亮红色故障灯；vApplicationStackOverflowHook() 中记录出错任务名后关闭 PWM 输出，令小车停机等待人工复位。

使用约束（必须注意）：

  • 通过 FreeRTOSConfig.h 的宏开关启用对应 Hook。
  • Hook 函数必须短小，避免长时间占用 CPU。
  • 在 Tick Hook 中不要调用可能阻塞的 API。
  • 在 Idle Hook 中通常也不应阻塞，否则会破坏空闲任务语义。

两个 Delay 函数​

vTaskDelay 与 vTaskDelayUntil 都会让任务进入 Blocked 状态，但一个基于“相对延时”，一个基于“绝对时刻”，这决定了它们的适用场景不同。

(1) vTaskDelay：相对延时​

  • 含义：从“当前调用时刻”开始，至少延时 xTicksToDelay 个 Tick 后再变为 Ready。
  • 特点：每轮循环执行时间会叠加到周期中，长期运行可能产生节拍漂移。
void vTaskDelay( const TickType_t xTicksToDelay );
/* xTicksToDelay: 延时的 Tick 数 */

适合场景：

  • 对周期抖动不敏感的普通轮询任务。
  • 仅需“让出 CPU 一段时间”的任务。
  • 示例：串口状态上报任务每次发送完一帧日志后调用 vTaskDelay(pdMS_TO_TICKS(200))，避免持续抢占 CPU，让同优先级任务有运行机会。

(2) vTaskDelayUntil：绝对周期延时​

  • 含义：阻塞到 (*pxPreviousWakeTime + xTimeIncrement) 这个绝对 Tick 时刻。
  • 特点：可将任务节拍锁定在固定周期，显著降低累计漂移。
void vTaskDelayUntil( TickType_t * const pxPreviousWakeTime,
                      const TickType_t xTimeIncrement );

说明：部分版本内核还提供 xTaskDelayUntil() 形式（返回 BaseType_t），本质用途与 vTaskDelayUntil() 一致。

参数说明：

  • pxPreviousWakeTime：上一次唤醒的基准时刻，首次使用前通常初始化为 xTaskGetTickCount()。
  • xTimeIncrement：周期增量（单位 Tick）。

适合场景：

  • 对固定采样周期有要求的任务（如传感器采样、控制闭环）。
  • 需要长期稳定节拍、希望减小累计时间漂移的周期任务。
  • 示例：IMU 采样任务要求 100Hz 固定频率运行，循环中使用 vTaskDelayUntil(&lastWakeTime, pdMS_TO_TICKS(10))，可避免“执行时间 + delay”叠加导致的周期漂移。

常见写法：

TickType_t lastWakeTime = xTaskGetTickCount();
for (;;)
{
   /* 周期任务逻辑 */
   vTaskDelayUntil(&lastWakeTime, pdMS_TO_TICKS(10));
}

(3) 选型结论​

  • 追求固定控制周期（采样、控制环）：优先 vTaskDelayUntil。
  • 只需简单延时让出 CPU：使用 vTaskDelay。
  • 不建议在任务中使用忙等延时（如空转循环或 HAL_Delay）替代 RTOS 延时接口。
