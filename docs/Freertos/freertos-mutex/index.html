互斥量

互斥量可以理解为“带钥匙的门”，同一时刻只能有一个任务拿着钥匙进入临界区。

  • 拿钥匙：take
  • 还钥匙：give

它和二值信号量看起来很像，但核心区别是：互斥量是专门为“保护共享资源”设计的，带 owner 语义和优先级继承机制。

使用场合​

适合使用互斥量的典型场景：

  • 多个任务共享同一个外设：UART、SPI、I2C、LCD、文件系统。
  • 多个任务操作同一份全局结构体或链表。
  • 需要避免“写一半被别人打断”的临界区。

不适合用互斥量的场景：

  • ISR 通知任务：应使用二值信号量或任务通知。
  • 只做简单事件同步（不涉及共享资源）：优先二值信号量。

一句话判断：你是在“等事件”还是“锁资源”？

  • 等事件：信号量/通知。
  • 锁资源：互斥量。

互斥量常用函数​

创建​

动态创建：

SemaphoreHandle_t xSemaphoreCreateMutex( void );
  • 成功返回句柄，失败返回 NULL。
  • 创建后初始状态是“可用”（相当于一把空闲钥匙）。

静态创建：

SemaphoreHandle_t xSemaphoreCreateMutexStatic( StaticSemaphore_t *pxMutexBuffer );
  • pxMutexBuffer：用户提供控制块内存。
  • 适合希望减少运行期动态内存分配的项目。

递归互斥量（可选）：

SemaphoreHandle_t xSemaphoreCreateRecursiveMutex( void );
SemaphoreHandle_t xSemaphoreCreateRecursiveMutexStatic( StaticSemaphore_t *pxMutexBuffer );
  • 同一任务可重复 take 多次，但必须对应 give 同样次数才真正释放。
  • 仅在确实存在“同任务嵌套加锁”需求时使用，避免滥用增加复杂度。

删除​

void vSemaphoreDelete( SemaphoreHandle_t xSemaphore );
  • 动态创建对象：删除后释放内核对象内存。
  • 静态创建对象：删除后对象失效，静态内存仍归用户管理。

take / give​

普通互斥量：

BaseType_t xSemaphoreTake( SemaphoreHandle_t xSemaphore,
                           TickType_t xTicksToWait );

BaseType_t xSemaphoreGive( SemaphoreHandle_t xSemaphore );

参数说明：

  • xSemaphore：互斥量句柄。
  • xTicksToWait：拿不到锁时最多等待多久。

返回值说明：

  • xSemaphoreTake：成功 pdTRUE，超时/失败 pdFALSE。
  • xSemaphoreGive：成功 pdTRUE，失败 pdFALSE（如 owner 不匹配）。

递归互斥量配套 API：

BaseType_t xSemaphoreTakeRecursive( SemaphoreHandle_t xMutex,
                                    TickType_t xTicksToWait );

BaseType_t xSemaphoreGiveRecursive( SemaphoreHandle_t xMutex );

重要约束：

  • 互斥量不能在 ISR 中使用（包括 take 和 give）。
  • 互斥量要求 owner 语义：通常谁拿锁，谁释放。

最小代码模板（任务间共享 UART）​

static SemaphoreHandle_t g_uartMutex;

void vTaskA(void *pv)
{
   for (;;)
   {
      if (xSemaphoreTake(g_uartMutex, pdMS_TO_TICKS(20)) == pdTRUE)
      {
         Uart_SendString("A task\r\n");
         xSemaphoreGive(g_uartMutex);
      }
      vTaskDelay(pdMS_TO_TICKS(50));
   }
}

void vTaskB(void *pv)
{
   for (;;)
   {
      if (xSemaphoreTake(g_uartMutex, pdMS_TO_TICKS(20)) == pdTRUE)
      {
         Uart_SendString("B task\r\n");
         xSemaphoreGive(g_uartMutex);
      }
      vTaskDelay(pdMS_TO_TICKS(50));
   }
}

void vAppInit(void)
{
   g_uartMutex = xSemaphoreCreateMutex();
   configASSERT(g_uartMutex != NULL);

   xTaskCreate(vTaskA, "A", 256, NULL, 2, NULL);
   xTaskCreate(vTaskB, "B", 256, NULL, 2, NULL);
}

这个模式的关键收益：

  • 同一时刻只有一个任务写 UART，日志不会互相打断。
  • 任务抢不到锁时可以阻塞等待，不会忙等烧 CPU。

为什么需要优先级继承​

先看“优先级反转”​

假设 3 个任务：

  • TaskL：低优先级，先拿到了互斥量。
  • TaskH：高优先级，随后想拿同一把锁，被阻塞。
  • TaskM：中优先级，不需要这把锁，但一直有活干。

如果没有优先级继承，可能出现：

  1. TaskL 拿锁进入临界区。
  2. TaskH 运行并请求锁，拿不到，进入阻塞。
  3. 调度器转去运行 TaskM（因为 TaskH 阻塞了，TaskM 比 TaskL 高）。
  4. TaskL 迟迟拿不到 CPU，没法尽快释放锁。
  5. 结果高优先级的 TaskH 被“间接卡住”。

这就是优先级翻转：高优先级任务被低优先级任务间接拖慢。

优先级继承怎么实现​

当 TaskH 因等待 TaskL 持有的互斥量而阻塞时：

  • 内核会临时把 TaskL 的优先级提升到不低于（一般就是等于）TaskH。
  • TaskL 更快获得 CPU，把临界区做完并释放锁。
  • 释放后，TaskL 优先级恢复原值，TaskH 拿到锁继续执行。

直观理解：让“占着钥匙的人”临时加速跑完，好尽快把钥匙交出来。

互斥量与临界区的选择​

两者都能保护共享资源，但侧重点不同：

对比项互斥量临界区（关中断/屏蔽调度）
等待方式可阻塞，让出 CPU调用方直接占住执行路径
保护范围任务间共享资源更偏底层短代码片段
实时性影响可控（设计得当）用不好会放大中断延迟
优先级继承有无

实践建议：

  • 业务层共享资源优先用互斥量。
  • 临界区用于非常短小、必须原子化的底层操作。

坏情况：递归加锁与死锁​

什么是递归加锁（递归锁）​

递归锁就是“同一任务可以重复拿同一把锁”的互斥量。

  • 第 1 次 take 成功后，owner 记录为当前任务，递归计数为 1。
  • 同一任务再次 take 同一把递归锁，仍会成功，递归计数加 1。
  • 只有当该任务把 give 次数执行到与 take 次数完全相等（计数归零）时，这把锁才真正释放给其他任务。

这和普通互斥量的关键区别是：

  • 普通互斥量：同一任务二次 take 会把自己阻塞（自我死锁）。
  • 递归互斥量：同一任务允许嵌套 take，用于分层调用链中的重复加锁场景。

递归互斥量 API：创建、获取与释放​

  1. xSemaphoreCreateRecursiveMutex()

    SemaphoreHandle_t xSemaphoreCreateRecursiveMutex( void );
    • 参数：无。
    • 返回值类型：SemaphoreHandle_t。
    • 返回值说明：成功返回递归互斥量句柄；失败返回 NULL（常见原因是内存不足）。
    • 说明：动态内存版本，控制块由 FreeRTOS 堆分配。

  2. xSemaphoreCreateRecursiveMutexStatic()

    SemaphoreHandle_t xSemaphoreCreateRecursiveMutexStatic(
        StaticSemaphore_t *pxMutexBuffer
    );
    • pxMutexBuffer（StaticSemaphore_t *）：用户提供的静态控制块内存地址。
    • 返回值类型：SemaphoreHandle_t。
    • 返回值说明：成功返回递归互斥量句柄；失败返回 NULL。
    • 说明：静态内存版本，不依赖运行期堆分配，适合对内存确定性要求高的工程。

  3. xSemaphoreTakeRecursive()

    BaseType_t xSemaphoreTakeRecursive(
        SemaphoreHandle_t xMutex,
        TickType_t xTicksToWait
    );
    • xMutex（SemaphoreHandle_t）：目标递归互斥量句柄。
    • xTicksToWait（TickType_t）：获取失败时的最大等待 Tick；0 表示不等待。
    • 返回值类型：BaseType_t。
    • 返回值说明：成功返回 pdTRUE；超时或失败返回 pdFALSE。
    • 说明：若当前任务已是 owner，再次获取会成功，递归计数加 1。

  4. xSemaphoreGiveRecursive()

    BaseType_t xSemaphoreGiveRecursive( SemaphoreHandle_t xMutex );
    • xMutex（SemaphoreHandle_t）：目标递归互斥量句柄。
    • 返回值类型：BaseType_t。
    • 返回值说明：成功返回 pdTRUE；失败返回 pdFALSE（常见于 owner 不匹配或释放时机错误）。
    • 说明：每调用一次递归计数减 1，只有减到 0 才真正释放给其他任务。

重要：递归互斥量必须用 TakeRecursive/GiveRecursive 成对调用，不要和普通 xSemaphoreTake/xSemaphoreGive 混用。

前提：FreeRTOSConfig.h 需要开启 configUSE_RECURSIVE_MUTEXES = 1。

什么是死锁​

在 FreeRTOS 里，死锁最常见的表现是：

  • 任务一直卡在 xSemaphoreTake(..., portMAX_DELAY)。
  • 系统不中断、不 HardFault，但某个业务功能“永远不再前进”。
  • 任务状态里能看到一个或多个任务长期 Blocked，且等待对象互相关联。

自我死锁（同一任务重复拿同一把普通互斥量）​

真实场景：串口驱动封装了发送锁。

  • 任务先调用 Uart_SendPacket()，内部 take(g_uartMutex) 成功。
  • Uart_SendPacket() 又调用 Uart_SendString()。
  • Uart_SendString() 里也执行 take(g_uartMutex)。
  • 因为普通互斥量不允许同一任务重复持有，第二次 take 会阻塞。
  • 但锁的 owner 正是它自己，这个任务已经被阻塞，永远走不到 give，形成自我死锁。

典型错误代码：

void Uart_SendString(const char *s)
{
   xSemaphoreTake(g_uartMutex, portMAX_DELAY);
   // ... 发送字符串
   xSemaphoreGive(g_uartMutex);
}

void Uart_SendPacket(const uint8_t *buf, uint16_t len)
{
   xSemaphoreTake(g_uartMutex, portMAX_DELAY);
   Uart_SendString("HEAD");   // 内部再次 take 同一把锁
   // ... 发送 buf
   xSemaphoreGive(g_uartMutex);
}

修复方法（按优先级推荐）：

  1. 统一加锁边界（首选）：只在最外层 API 加锁，内部函数默认“调用者已持锁”。
  2. 拆分接口：提供 Uart_SendStringLocked()（不加锁）和 Uart_SendString()（外部入口加锁）。
  3. 确有嵌套需求时使用递归互斥量：xSemaphoreCreateRecursiveMutex() + xSemaphoreTakeRecursive() / xSemaphoreGiveRecursive()，并保证 take/give 次数严格配平。

任务间循环等待死锁（A 等 B，B 等 A）​

真实场景：日志模块和存储模块各有一把锁。

  • 任务 A：先拿 logMutex，再等 flashMutex。
  • 任务 B：先拿 flashMutex，再等 logMutex。
  • 两个任务都拿着一把锁等待另一把，进入循环等待。

规避方法：

  • 全项目统一锁顺序（例如“总是先 flashMutex，后 logMutex”）。
  • 持锁时不做阻塞等待（尤其是 portMAX_DELAY）。
  • 使用有限超时并做失败回退，避免“永远等待”：
if (xSemaphoreTake(flashMutex, pdMS_TO_TICKS(20)) != pdTRUE)
{
   // 记录失败并回退，稍后重试
   return;
}

FreeRTOS 里“拯救死锁”的常用手段​

  • 预防优先于补救：统一锁顺序、缩短持锁时间、避免嵌套锁。
  • 对关键 take 使用超时，不要全都 portMAX_DELAY。
  • 启用诊断信息：
    • configUSE_TRACE_FACILITY + vTaskGetInfo() 查看任务长期阻塞点。
    • 在失败分支打印“当前任务名 + 锁名 + 等待时长”。
  • 结构性优化：把共享外设改成“单服务任务 + 队列请求”，从根上减少多锁竞争。

结论：死锁通常不是“单个 API 写错”，而是锁粒度、锁顺序、调用边界三者没有统一约束。

互斥量与队列在共享资源管理中的对比​

对比项互斥量（Mutex）队列（Queue）
核心用途保护临界资源，保证同一时刻单任务访问解耦生产者和消费者，传递请求/数据
是否直接传数据否是（按项拷贝）
典型模式多任务竞争同一外设时加锁访问建立“服务任务 + 请求队列”串行化访问
优先级反转处理支持优先级继承不适用（通过消息流避免直接竞争）
何时优先选择资源访问路径短、持锁时间可控冲突高、访问耗时长、希望彻底去共享化

结论：二者不是互斥替代关系。低冲突短临界区优先互斥量；高冲突或长耗时资源优先“服务任务 + 队列”。

工程建议与易错点​

  1. 临界区只放“必须原子化”的语句，避免使用耗时函数，进行耗时操作。 真实情况：持锁后调用 printf、HAL_UART_Transmit（长超时）会把锁占住几毫秒到几十毫秒，其他任务全部排队。

  2. 禁止“持有锁时再等待另一把锁”。 真实情况：协议栈任务拿 netMutex 后等待 flashMutex，存储任务反过来，现场就是经典环路等待。 建议：在设计文档里明确全局锁顺序，代码里按顺序执行。

  3. 持锁期间不要做长阻塞操作。 典型高风险调用：

    • 网络发送/接收（链路抖动会拉长等待）
    • 慢速外设访问（I2C EEPROM、Flash 擦写）
    • 大量日志输出（串口带宽不足时阻塞）

  4. ISR 中不要使用互斥量，应使用对应的 ISR API。 正确做法：ISR 用 xSemaphoreGiveFromISR、vTaskNotifyGiveFromISR 发事件，由任务上下文去拿互斥量并访问共享资源。

  5. 高冲突资源优先考虑“服务任务化”。 真实情况：多个任务同时访问 SPI Flash，锁竞争严重、时延不可控。 改造方式：建立 FlashServiceTask，其他任务通过队列提交读写请求；资源只在服务任务内部访问，不再需要多任务争锁。

实践目标：不是“彻底不用锁”，而是让锁的等待可预期、故障可定位、系统可恢复。
